# 02 数据流设计

> 描述七步闭环流程中各环节的数据输入、处理逻辑、输出及流转关系，明确AI/GIS服务调用时机与数据存储位置。

---

## 一、数据流总览

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ ① 计划   │───→│ ② 执行   │───→│ ③ 发现   │───→│ ④ 工单   │───→│ ⑤ 处置   │───→│ ⑥ 验收   │───→│ ⑦ 分析   │
│ 编排下发  │    │ 现场巡检  │    │ 上报识别  │    │ 创建分派  │    │ 处置反馈  │    │ 结果确认  │    │ 数据决策  │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    └────┬────┘
     ↑                                                                                          │
     └────────────────────────────────── 数据回流自优化 ──────────────────────────────────────────┘
```

## 二、分环节数据流

### 2.1 ① 计划 — 巡检计划编排与下发

| 维度 | 内容 |
|------|------|
| **触发条件** | 每日06:00系统自动触发；管理层手动调整触发 |
| **输入数据** | ① 历史巡检记录（近30天）；② 园区矢量地图（边界/楼栋/楼层）；③ 巡检点位清单；④ 上期分析报告的优化建议；⑤ 人员排班表 |
| **处理逻辑** | 1. 读取历史数据 → 2. AI预测分析计算各区域问题频率 → 3. GIS覆盖分析验证路线盲区 → 4. 生成当日计划 → 5. 推送至移动端 |
| **输出数据** | 当日巡检计划（路线序列、点位清单、checklist模板、预计时长） |
| **下一环节** | ② 执行（巡检员接收计划并开始巡检） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 计划生成时 | 预测分析 | 历史问题分布、巡检频次 | 推荐巡检频次（升频/降频/保持） | 存入计划建议日志 |
| 计划生成时 | 路线优化 | 点位坐标、历史耗时 | 最优路线序列 | 作为计划字段存储 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 计划生成时 | 覆盖分析 | 园区边界Polygon、路线Polyline、点位集合 | 盲区区域列表（如有） | 存入覆盖分析记录 |
| 计划展示时 | 空间可视化 | 路线数据、点位数据 | 地图渲染数据（GeoJSON） | 不持久化，实时计算 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 巡检计划 | 业务数据库 — `inspection_plan` 表 | 含路线ID、点位序列、分配人员、计划时间 |
| 路线定义 | 业务数据库 — `route` 表 + GIS引擎 | 路线节点坐标序列 |
| 覆盖分析结果 | 业务数据库 — `coverage_report` 表 | 盲区记录、覆盖率百分比 |
| 历史巡检记录 | 业务数据库 — `inspection_record` 表 | 用于AI预测的回溯数据 |

### 2.2 ② 执行 — 现场巡检执行

| 维度 | 内容 |
|------|------|
| **触发条件** | 巡检员点击"开始巡检" |
| **输入数据** | ① 当日巡检计划；② 实时GPS坐标；③ 巡检员操作事件（到达点位、完成checklist、拍照） |
| **处理逻辑** | 1. 加载计划 → 2. 启动GPS采样 → 3. 逐点导航+checklist检查 → 4. 实时同步进度 → 5. 异常行为检测 |
| **输出数据** | 巡检轨迹、点位完成记录、现场照片/视频、进度状态 |
| **下一环节** | ③ 发现（发现问题时上报）或 ⑧ 分析（无问题完成巡检） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 执行过程中 | 异常检测 | checklist完成时间间隔、点击模式 | 是否疑似跳过检查的预警 | 存入行为日志 |
| 接近高频问题区 | 实时预警 | 当前坐标、历史问题热区 | 提示"该区域近7天出现X次问题" | 不存储，实时推送 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 每10秒 | 轨迹记录 | GPS坐标（lat/lng/alt/timestamp） | 轨迹点序列 | 存入轨迹数据库 |
| 到达点位 | 定位验证 | 当前坐标、点位坐标 | 距离是否≤20m | 存入点位打卡记录 |
| 导航切换 | 路径规划 | 当前位置、下一点位 | 导航路径（GeoJSON） | 不存储，实时计算 |
| 实时展示 | 位置同步 | 当前坐标 | 实时位置Marker | 不存储，WebSocket推送 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| GPS轨迹 | 业务数据库 — `trajectory` 表 | 点序列（lat, lng, timestamp, accuracy） |
| 点位完成记录 | 业务数据库 — `checkpoint_record` 表 | 点位ID、完成时间、照片URL、检查结果 |
| 现场照片/视频 | 文件存储 — OSS/MinIO | 按日期+巡检员ID分目录 |
| 行为日志 | 业务数据库 — `behavior_log` 表 | 用于AI异常检测回溯 |

### 2.3 ③ 发现 — 问题发现与上报

| 维度 | 内容 |
|------|------|
| **触发条件** | 巡检员点击"上报问题" |
| **输入数据** | ① 现场照片（1-多张）；② 语音/文字描述；③ 当前GPS坐标；④ 点位编号；⑤ 巡检员ID |
| **处理逻辑** | 1. 接收多媒体 → 2. AI图像识别问题类型 → 3. NLP结构化描述 → 4. 自动附加时空信息 → 5. 生成上报记录 |
| **输出数据** | 问题上报记录（含问题类型、严重等级预判定、结构化工单字段、定位信息） |
| **下一环节** | ④ 分支判断（系统自动判定严重程度并选择路径） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 照片上传后 | 图像识别 | JPEG/PNG图像 | 问题类型（如"消防通道堆物"）、置信度 | 存入识别结果表 |
| 语音录入后 | NLP结构化 | 语音转文字后的文本 | 提取：问题类型、严重程度词、影响范围 | 存入结构化字段 |
| 置信度<85% | 人工确认 | 识别结果+原始数据 | 巡检员修正后的最终类型 | 更新识别结果表 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 上报提交时 | 定位打标 | GPS坐标、点位关联规则 | 地图标记坐标、所属区域/楼栋/楼层 | 存入问题记录的空间字段 |
| GPS精度不足 | 手动校准 | 用户拖拽后的坐标 | 修正后的标记位置 | 更新问题坐标 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 问题上报记录 | 业务数据库 — `issue_report` 表 | 问题ID、类型、描述、坐标、时间、上报人 |
| 原始照片 | 文件存储 | 原图保留，用于后续比对 |
| AI识别结果 | 业务数据库 — `ai_recognition` 表 | 模型版本、识别类型、置信度、是否人工修正 |
| NLP结构化结果 | 业务数据库 — `issue_report` 表（JSON字段） | 结构化后的工单预填字段 |
| 问题标记图层 | GIS引擎 + 业务数据库 | 问题坐标索引，支持空间查询 |

### 2.4 ④ 工单 — 工单创建与分派

| 维度 | 内容 |
|------|------|
| **触发条件** | 分支判断为"严重"后自动触发；或人工创建 |
| **输入数据** | ① 步骤③的上报记录（含AI识别结果）；② 处置人档案（技能标签/历史效率/当前负载）；③ 处置人实时坐标；④ SLA规则配置 |
| **处理逻辑** | 1. 自动创建工单（预填字段） → 2. AI推荐处置人 → 3. GIS计算就近距离 → 4. 管理层确认分派 → 5. 推送通知至处置人 |
| **输出数据** | 已分派工单（含处置人、SLA时限、导航入口） |
| **下一环节** | ⑤ 处置（处置人接单并处理问题） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 工单创建时 | 严重程度预测 | 图像识别结果、问题类型、历史数据 | P0/P1/P2/P3等级 | 存入工单字段 |
| 分派前 | 智能推荐 | 问题类型、处置人技能、位置、负载、历史效率 | Top3处置人候选+推荐理由 | 存入推荐日志 |
| 工单创建时 | 工单结构化 | NLP提取的字段 | 标准化工单：问题类型/位置/影响范围/紧急度 | 作为工单字段存储 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 推荐处置人时 | 就近匹配 | 问题点位坐标、各处置人实时坐标 | 步行/驾车距离排序 | 参与推荐排序权重计算 |
| 管理层分派时 | 空间可视化 | 问题点位 + 候选处置人位置 | 地图上同时展示点位和人 | 不存储，实时渲染 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 工单主记录 | 业务数据库 — `work_order` 表 | 工单ID、状态、问题ID、处置人ID、SLA时限 |
| 工单快照 | 业务数据库 — `work_order_snapshot` 表 | 创建时的上报数据快照，防篡改 |
| 推荐结果 | 业务数据库 — `dispatch_recommendation` 表 | AI推荐记录，用于效果评估 |
| 分派记录 | 业务数据库 — `dispatch_record` 表 | 分派人、分派时间、处置人确认时间 |

### 2.5 ⑤ 处置 — 问题处置与反馈

| 维度 | 内容 |
|------|------|
| **触发条件** | 处置人点击"开始处置" |
| **输入数据** | ① 已分派工单；② 处置人当前坐标；③ AI推荐的处置方案；④ 处置过程中的照片/文字描述 |
| **处理逻辑** | 1. 导航到场 → 2. 到场打卡（GPS验证） → 3. 按方案处置 → 4. 上传处置结果 → 5. 提交完成 |
| **输出数据** | 处置完成记录（到场时间、处置描述、处置后照片、耗时） |
| **下一环节** | ⑥ 验收（触发验收流程） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 处置开始时 | 知识库推荐 | 问题类型、历史工单库 | 相似工单的处置方案、平均耗时 | 展示给处置人，不强制存储 |
| 到场时 | SLA验证 | 分派时间、到场时间、SLA规则 | 是否超时、超时等级 | 存入处置记录 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 接单后 | 导航指引 | 处置人当前位置、问题点位 | 导航路径 | 不存储，实时计算 |
| 到场打卡 | 位置验证 | 当前GPS坐标、问题点位坐标 | 距离≤50m判定为有效到场 | 存入到场记录 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 处置记录 | 业务数据库 — `work_order_action` 表 | 到场时间、处置描述、完成时间、耗时 |
| 处置后照片 | 文件存储 | 用于后续AI比对验收 |
| 到场验证记录 | 业务数据库 — `arrival_verification` 表 | GPS坐标、验证距离、验证结果 |

### 2.6 ⑥ 验收 — 处置结果验收

| 维度 | 内容 |
|------|------|
| **触发条件** | 处置人提交完成后自动触发 |
| **输入数据** | ① 上报时照片（步骤③）；② 处置后照片（步骤⑤）；③ 处置描述；④ 验收人GPS坐标（现场验收时） |
| **处理逻辑** | 1. AI图像比对 → 2. 质量评分 → 3. 验收人确认/复核 → 4. 判定通过/不通过 |
| **输出数据** | 验收结果（通过/不通过）、AI比对置信度、质量评分 |
| **下一环节** | 不通过 → 退回⑤重新处置；通过 → ⑦ 分析 |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 处置提交后 | 图像比对 | 上报照片 vs 处置后照片 | 修复判定结果、置信度评分 | 存入验收记录 |
| 验收完成后 | 质量评分 | 图像比对结果、处置描述完整性、到场时效 | 综合质量分（1-5分） | 存入处置人绩效参考 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 现场验收时 | 现场确认 | 验收人GPS坐标、问题点位坐标 | 距离≤20m判定有效 | 存入验收记录 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 验收记录 | 业务数据库 — `acceptance_record` 表 | 验收人、验收时间、结果、AI置信度、质量分 |
| 比对结果 | 业务数据库 — `image_comparison` 表 | 比对算法版本、置信度、判定依据 |

### 2.7 ⑦ 分析 — 数据分析与决策优化

| 维度 | 内容 |
|------|------|
| **触发条件** | 每日22:00自动触发；管理层手动触发 |
| **输入数据** | ① 全流程数据（计划/轨迹/问题/工单/处置/验收）；② 空间数据（问题点位集合）；③ 时间序列数据 |
| **处理逻辑** | 1. 汇总当日数据 → 2. AI趋势预测+异常检测 → 3. GIS热力图聚类 → 4. 生成优化建议 → 5. 输出报告 |
| **输出数据** | 日报/周报/月报、问题热力图、趋势预测、优化建议 |
| **下一环节** | 数据回流至 ① 计划（管理层确认后自动更新巡检计划） |

**AI服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 报告生成时 | 趋势预测 | 历史问题时间序列、区域维度 | 未来7天各区域问题概率预测 | 存入预测结果表 |
| 报告生成时 | 异常检测 | 巡检完成率、问题数量、漏检率时序 | 异常模式告警（如突增50%） | 存入异常告警表 |
| 报告生成时 | 优化建议 | 趋势预测+异常检测结果 | 频次/路线/点位/人员调整建议 | 存入优化建议表 |

**GIS服务调用：**

| 调用时机 | 服务 | 输入 | 输出 | 存储 |
|---------|------|------|------|------|
| 报告生成时 | 热力图聚类 | 历史问题点位集合（近30天） | DBSCAN聚类结果、热力图图层 | 存入聚类结果表 |
| 报告生成时 | 覆盖率分析 | 巡检轨迹集合、园区边界 | 覆盖率百分比、盲区列表 | 存入覆盖率报告 |

**数据存储：**

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 日报/周报/月报 | 业务数据库 — `report` 表 | 报告类型、生成时间、核心指标JSON |
| 热力图聚类结果 | 业务数据库 — `cluster_result` 表 | 聚类中心坐标、半径、问题数量、密度等级 |
| 预测结果 | 业务数据库 — `prediction` 表 | 预测日期、区域、概率值、模型版本 |
| 优化建议 | 业务数据库 — `optimization_suggestion` 表 | 建议类型、触发条件、建议内容、管理层确认状态 |
| 回流确认记录 | 业务数据库 — `plan_adjustment` 表 | 确认时间、调整内容、生效时间 |

## 三、数据存储总览

### 3.1 按存储类型划分

| 存储类型 | 存储内容 | 技术选型 [假设] |
|---------|---------|----------------|
| **关系型数据库** | 用户、计划、工单、验收、报告等业务实体 | PostgreSQL |
| **GIS空间数据库** | 轨迹、点位、问题标记、路线、聚类结果（PostGIS扩展） | PostgreSQL + PostGIS |
| **文件存储** | 照片、语音、视频、报告导出文件 | OSS / MinIO |
| **缓存** | 实时位置、会话状态、热点数据 | Redis |
| **时序数据（V2.0）** | IoT传感器数据、高频GPS采样 | InfluxDB/TDengine |

### 3.2 按数据主题划分

| 数据主题 | 核心表/集合 | 数据来源 | 消费方 |
|---------|-----------|---------|--------|
| **巡检计划** | `inspection_plan`, `route`, `checkpoint` | ①计划环节生成 | ②执行环节读取 |
| **轨迹数据** | `trajectory`, `checkpoint_record` | ②执行环节采集 | ⑦分析环节统计 |
| **问题数据** | `issue_report`, `ai_recognition` | ③发现环节上报 | ④工单环节引用 |
| **工单数据** | `work_order`, `dispatch_record`, `work_order_action` | ④⑤环节产生 | ⑥验收环节读取 |
| **验收数据** | `acceptance_record`, `image_comparison` | ⑥验收环节产生 | ⑦分析环节统计 |
| **分析数据** | `report`, `cluster_result`, `prediction`, `optimization_suggestion` | ⑦分析环节产出 | ①计划环节回流 |
| **空间数据** | GIS引擎图层、空间索引 | GIS引擎维护 | 全环节调用 |
| **AI模型数据** | 模型版本、识别日志、推荐日志 | AI引擎维护 | 用于模型效果评估 |

## 四、跨环节数据血缘

## 六、数据安全与合规

| 数据类型 | 处理要求 |
|---------|---------|
| **巡检轨迹** | 仅保存90天轨迹明细，超期聚合为日报统计值后删除原始点 |
| **现场照片** | 原图保留1年，用于AI模型训练和争议回溯；公开报告使用脱敏缩略图 |
| **语音数据** | 转文字后删除原始音频，除非用户明确保留 |
| **个人位置** | 实时位置仅管理层可见，接口增加范围权限控制（只能看自己管辖区域人员） |
| **AI识别结果** | 人工修正记录保留，用于模型迭代和效果评估 |

## 七、数据流设计约束

| 约束项 | 说明 |
|--------|------|
| **不做实时数仓** | V1.0分析报表基于业务数据库定时任务生成，实时数仓为V2.0扩展 |
| **不做数据湖** | V1.0非结构化数据（照片/语音）按文件存储管理，不进入数据湖 |
| **接口级别不展开** | 本数据流设计定义数据主题和流转关系，不定义具体API契约 |
| **单园区数据隔离** | V1.0按单园区设计，数据层面不做多租户隔离；V2.0扩展时增加园区ID字段 |
