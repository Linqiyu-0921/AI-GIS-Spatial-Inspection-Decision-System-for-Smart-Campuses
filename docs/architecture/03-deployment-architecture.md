# 03 部署架构设计

> 描述智慧园区空间巡检决策系统的运行时部署结构，明确各层部署位置、外部服务依赖及版本演进规划。

## 一、部署架构总览

系统采用经典的三层部署结构：客户端层 → 服务层 → 数据层。V1.0为单体应用部署，不做容器编排和微服务拆分。

## 二、各层部署说明

### 2.1 客户端层

| 客户端 | 部署形态 | 技术栈 [假设] | 核心能力 | 版本 |
|--------|---------|--------------|---------|------|
| **移动端** | iOS/Android 原生App | Flutter / React Native | GPS定位、相机拍照、离线缓存、推送通知 | V1.0 |
| **PC端** | Web单页应用（SPA） | Vue 3 / React | GIS地图渲染、数据可视化、工单管理 | V1.0 |
| **大屏端（V2.0）** | 独立大屏应用 | DataV / 自建大屏 | 园区实时态势一屏览 | V2.0 |

**部署说明：**
- 移动端通过应用商店/企业分发部署，支持离线巡检后增量同步
- PC端为Web应用，部署在Nginx静态服务器上，通过HTTPS访问
- V1.0不做桌面客户端，V2.0视需求评估Electron方案

---

### 2.2 服务层

#### 2.2.1 API网关

| 属性 | 说明 |
|------|------|
| **部署形态** | 单体应用入口，与业务服务同进程部署（V1.0） |
| **职责** | 统一认证鉴权、路由分发、请求限流、日志记录 |
| **技术栈 [假设]** | Spring Cloud Gateway / Nginx + 自研网关 |
| **部署数量** | 2节点负载均衡（主备），V2.0演进为独立网关集群 |

#### 2.2.2 AI服务层

| 服务 | 部署形态 | 外部依赖（V1.0） | 自研/第三方 | V2.0规划 |
|------|---------|-----------------|-----------|---------|
| **图像识别API** | 独立服务进程 | 阿里云视觉智能（通用图像分类、自定义模型训练） | 第三方（V1.0）→ 自研（V2.0） | 迁移至自研模型，支持视频流识别 |
| **NLP结构化API** | 独立服务进程 | 阿里云NLP / 百度智能云（语音转写、信息抽取） | 第三方（V1.0）→ 混合（V2.0） | 引入自研领域模型，提升园区专业术语识别率 |
| **预测分析API** | 独立服务进程 | 阿里云PAI / 自研Python服务（scikit-learn/xgboost） | 混合（V1.0）→ 自研（V2.0） | 模型自训练平台，支持AutoML调优 |

**部署说明：**
- V1.0 AI服务以独立进程部署在同一服务器内，通过内网HTTP调用
- 图像识别和NLP在V1.0以第三方云服务为主，降低算法团队投入
- 预测分析因业务逻辑耦合度高，V1.0即采用自研Python服务部署
- 各AI服务预留模型版本管理接口，支持灰度切换

#### 2.2.3 GIS服务层

| 服务 | 部署形态 | 外部依赖（V1.0） | 自研/第三方 | V2.0规划 |
|------|---------|-----------------|-----------|---------|
| **地图渲染API** | 独立服务进程 | 高德地图API（2D矢量底图、瓦片服务） | 第三方 | 保持不变，V2.0叠加BIM图层 |
| **空间分析API** | 独立服务进程 | PostGIS（空间计算）+ 自研Java/Go服务 | 混合 | 扩展3D空间分析能力 |
| **路径规划API** | 独立服务进程 | 高德地图API（步行/驾车路径规划） | 第三方（V1.0）→ 混合（V2.0） | V2.0引入室内路径规划自研模块 |

**部署说明：**
- 地图渲染和路径规划在V1.0依赖高德地图API，通过公网HTTPS调用
- 空间分析基于PostGIS扩展，部署在数据库服务器同机房，降低网络延迟
- GIS服务缓存热点瓦片和路径结果至Redis，减少第三方API调用频次

---

### 2.3 数据层

| 数据存储 | 部署形态 | 技术选型 [假设] | 存储内容 | 版本 |
|---------|---------|----------------|---------|------|
| **业务数据库** | 主从部署（1主1从） | PostgreSQL 14+ | 用户、计划、工单、验收、报告等业务实体 | V1.0 |
| **空间数据库** | 与业务数据库同实例，PostGIS扩展 | PostgreSQL + PostGIS 3.1+ | 轨迹、点位、问题标记、路线、聚类结果 | V1.0 |
| **文件存储** | 对象存储服务 | 阿里云OSS / MinIO | 照片、语音、视频、报告导出 | V1.0 |
| **缓存层** | 单节点/哨兵模式 | Redis 6+ | 会话状态、实时位置、热点数据、地图瓦片缓存 | V1.0 |
| **时序数据库（V2.0）** | 独立集群 | InfluxDB / TDengine | IoT传感器数据、高频GPS采样 | V2.0 |
| **BIM存储（V2.0）** | 文件存储+元数据库 | 自研 + PostgreSQL | 3D建筑模型、设备空间坐标 | V2.0 |

**部署说明：**
- V1.0数据库采用主从架构，主库写、从库读，每日全量备份+实时binlog
- 空间数据与业务数据共用PostgreSQL实例，通过Schema隔离（`public` vs `gis`）
- 文件存储使用阿里云OSS（生产环境）或MinIO（私有化部署）
- Redis部署哨兵模式保障高可用，缓存过期策略：轨迹实时位置5分钟、会话30分钟、热点数据1小时

## 五、部署约束与决策

| 约束项 | 决策 | 说明 |
|--------|------|------|
| **不做K8s/Docker** | V1.0采用传统虚拟机/物理机部署 | 降低运维复杂度，V2.0评估容器化收益后决策 |
| **不做微服务拆分** | V1.0单体应用，AI/GIS服务以独立进程部署在同一服务器 | 减少网络调用开销，V2.0按领域拆分微服务 |
| **单数据中心** | V1.0单机房部署，不做异地多活 | MVP阶段成本优先，V2.0评估灾备需求 |
| **AI服务优先第三方** | V1.0图像识别和NLP依赖阿里云等第三方云服务 | 降低算法团队投入，快速验证业务价值 |
| **GIS服务混合部署** | 地图渲染和路径规划用第三方API，空间分析用自研+PostGIS | 平衡成本和性能，核心空间计算自持 |
| **数据库主从架构** | PostgreSQL一主一从，满足V1.0读写需求 | 简单可靠，V2.0评估读写分离中间件 |

## 六、版本演进路线

### 6.1 V1.0 部署形态（当前）

- **应用层**：单体Java/Node.js应用，2节点负载均衡
- **AI层**：图像识别+NLP调用第三方云API，预测分析自研Python服务本地部署
- **GIS层**：地图渲染+路径规划调用高德API，空间分析自研服务+PostGIS
- **数据层**：PostgreSQL主从、Redis单节点/哨兵、OSS文件存储
- **运维**：传统虚拟机部署，手动发布，日志本地存储

### 6.2 V2.0 部署演进（规划）

| 演进项 | V1.0 | V2.0 |
|--------|------|------|
| **应用架构** | 单体应用 | 微服务拆分（计划/执行/工单/分析服务独立部署） |
| **容器化** | 虚拟机部署 | 评估Docker容器化，视团队能力决策K8s |
| **AI服务** | 第三方为主 | 自研模型服务化，引入模型版本管理和A/B测试 |
| **GIS服务** | 2D+第三方API | 3D BIM渲染自研，室内定位服务接入 |
| **数据层** | 关系型为主 | 引入时序数据库（InfluxDB/TDengine）、BIM模型存储 |
| **DevOps** | 手动发布 | CI/CD流水线，自动化测试，灰度发布 |
| **监控告警** | 基础日志 | 引入APM（应用性能监控）、链路追踪、业务指标监控 |

## 七、部署架构设计约束

| 约束项 | 说明 |
|--------|------|
| **不做运维级别部署图** | 本部署架构聚焦应用层和数据层的运行时结构，不展开网络拓扑、安全组、防火墙规则 |
| **不做容量规划** | 服务器规格和数量根据实际用户规模在实施阶段确定，本文档不做精确容量测算 |
| **技术选型为示意** | 所有技术栈标注 `[假设]`，实际实施可根据团队技术储备调整 |
| **单园区部署** | V1.0按单园区单套环境设计，多园区部署为V2.0扩展 |
