## 三、AI 嵌入点分布

系统围绕 **"计划 → 执行 → 处置 → 优化"** 巡检业务闭环，在 7 个环节嵌入 AI 能力：

| 序号 | 巡检环节 | 嵌入模块 | AI 能力 | 输入数据 | 输出数据 |
|------|----------|----------|---------|---------|---------|
| 1 | **计划制定** | 智能推荐模块 | 基于历史数据的巡检路线/周期推荐 | 历史巡检记录、问题频次统计、建筑平面图 | 推荐巡检路线、建议巡检周期、优先级排序 |
| 2 | **现场巡检** | 图像识别模块 | 拍照自动识别问题类别（裂缝/漏水/设备故障） | 现场照片 | 问题类别、置信度评分、异常区域标注 |
| 3 | **现场巡检** | 语音交互模块 | 语音描述转写 + 关键信息结构化提取 | 语音描述音频 | 文字转写结果、结构化问题描述（位置/类型/严重程度） |
| 4 | **问题记录** | 智能分类模块 | 基于图像+NLP的多模态问题分类与严重程度评估 | 照片+语音+位置信息 | 标准化问题记录（类别/等级/关联设备） |
| 5 | **工单分派** | 智能派单模块 | 基于工单类型、人员技能、地理位置的最优分派 | 新工单信息、维修人员状态、地理距离 | 推荐处理人、预计响应时间、优先级建议 |
| 6 | **趋势分析** | 预测分析模块 | 基于历史数据的问题趋势预测与异常检测 | 历史问题记录、设备台账、环境数据 | 趋势预测报告、异常告警、维护建议 |
| 7 | **巡检优化** | 智能优化模块 | 基于巡检效果反馈的计划自适应调整 | 巡检完成率、问题发现率、重复问题率 | 优化建议、路线调整方案、频次调整建议 |

## 四、GIS 嵌入点分布

系统围绕 **"定位 → 路线 → 标记 → 分析 → 可视化"** 地理空间能力链，在 7 个环节嵌入 GIS 能力：

| 序号 | 巡检环节 | 嵌入模块 | GIS 能力 | 输入数据 | 输出数据 |
|------|----------|----------|---------|---------|---------|
| 1 | **位置定位** | LBS服务模块 | GPS实时定位 + 地理围栏检测 | GPS信号、基站/WiFi信号 | 经纬度坐标、所在区域/楼栋判定、地理围栏触发状态 |
| 2 | **路线规划** | 路径规划模块 | 最优巡检路线计算、导航引导 | 起点、巡检点列表、通行约束 | 推荐路线、预计通行时间、导航指引 |
| 3 | **轨迹记录** | 轨迹管理模块 | GPS轨迹实时记录、历史轨迹存储与回放 | GPS定位流 | 轨迹线串、巡检覆盖范围、时间轴同步 |
| 4 | **点位标记** | 地图标记模块 | 问题点位空间标记、多媒体附件关联 | 问题记录、照片、经纬度 | 地图标注点、弹窗详情、聚合展示 |
| 5 | **空间查询** | 空间分析模块 | 基于地理范围的工单/问题统计、周边资源查询 | 矩形/多边形区域、缓冲区 | 区域内问题统计、热力分布、周边维修资源 |
| 6 | **热力可视化** | 热力图模块 | 问题密度热力图、巡检频率热力图 | 问题点位集合、时间窗口 | 热力图层、密度分级、时间序列动画 |
| 7 | **大屏展示** | GIS地图大屏 | 2.5D/3D园区地图、多图层叠加、实时态势展示 | 园区底图、实时数据流 | 交互式地图大屏、图层切换、数据联动 |

### 5.2 各环节数据流详情

#### 环节 1：计划创建

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 管理员在PC端手动创建 / 系统按模板自动生成（V2.0：AI推荐） |
| **输入数据** | 巡检类型、区域范围、执行周期、巡检点位、执行人、关联模板 |
| **业务逻辑** | 1. 参数校验与重复检查 → 2. 根据巡检点位自动生成推荐路线 → 3. 检查执行人排班冲突 → 4. 保存计划并生成初始任务 |
| **输出数据** | 巡检计划记录（plan）、关联任务列表（task_list）、推荐路线数据 |
| **下一环节** | 环节 2：任务派发 |
| **AI 服务调用** | 智能路线推荐（V1.0 基础规则，V2.0 强化学习优化） |
| **GIS 服务调用** | 路线规划 API → 高德路径规划接口 |
| **数据存储** | `inspection_plan` → PostgreSQL（业务数据库）；巡检路线 → PostGIS（空间数据库） |

#### 环节 2：任务派发

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 计划定时触发 / 管理员手动派发 / 自动派发规则匹配 |
| **输入数据** | 计划 ID、可用执行人列表、紧急任务、执行人实时位置 |
| **业务逻辑** | 1. 根据计划生成具体任务 → 2. 匹配执行人（技能+地理位置+工作量） → 3. 推送任务通知 → 4. 执行人确认接收 |
| **输出数据** | 任务记录（task）、派发记录、通知消息 |
| **下一环节** | 环节 3：现场巡检 |
| **AI 服务调用** | 智能派单推荐：基于执行人技能/位置/工作量的最优匹配 |
| **GIS 服务调用** | 就近匹配计算：执行人位置与任务区域的距离矩阵 |
| **数据存储** | `inspection_task` → PostgreSQL；执行人实时位置 → Redis（缓存层） |

#### 环节 3：现场巡检

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 执行人在移动端点击"开始巡检" |
| **输入数据** | 任务 ID、GPS 定位流、现场照片、语音描述 |
| **业务逻辑** | 1. GPS定位验证是否在巡检区域内 → 2. 按路线引导执行人逐点巡检 → 3. 每个点位拍照+语音/文字记录 → 4. AI辅助识别问题 → 5. 提交巡检记录 |
| **输出数据** | 巡检记录（inspection_record）、GPS 轨迹、现场照片/语音、AI 识别结果 |
| **下一环节** | 环节 4：问题记录 |
| **AI 服务调用** | 1. 图像识别：照片 → 问题类别+置信度；2. NLP结构化：语音 → 文字 → 结构化问题描述 |
| **GIS 服务调用** | 1. 地理围栏检测：GPS → 是否在巡检区域内；2. 轨迹记录：GPS流 → 轨迹线串 |
| **数据存储** | `inspection_record` → PostgreSQL；GPS轨迹 → PostGIS（LineString）；照片/语音 → OSS |

#### 环节 4：问题记录

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 巡检过程中发现问题 / AI自动标记异常 / 执行人手动添加 |
| **输入数据** | 关联巡检记录、问题描述、照片证据、严重程度评估、GPS坐标 |
| **业务逻辑** | 1. AI多模态分类（图像+NLP）→ 2. 严重程度自动评估 → 3. 关联设备/资产信息 → 4. 生成标准化问题记录 → 5. 触发工单创建（严重问题自动创建） |
| **输出数据** | 问题记录（issue）、关联证据文件、严重等级评估、自动工单（如触发） |
| **下一环节** | 环节 5：工单处置 |
| **AI 服务调用** | 1. 智能分类：图像+NLP → 问题类别+严重程度；2. 设备关联：问题描述 → 关联设备台账 |
| **GIS 服务调用** | 点位标记：问题GPS坐标 → 地图标注点 |
| **数据存储** | `issue` → PostgreSQL；问题点位 → PostGIS（Point）；证据文件 → OSS |

#### 环节 5：工单处置

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 问题记录自动触发 / 手动创建工单 / 领导指派 |
| **输入数据** | 问题记录 ID、维修人员信息、处理方案、材料消耗 |
| **业务逻辑** | 1. AI智能派单（V1.0 基础规则，V2.0 强化学习） → 2. 维修人员接收并响应 → 3. 现场处理+拍照记录 → 4. 处理结果提交 → 5. 质量抽检（V2.0：AI自动质检） |
| **输出数据** | 工单记录（work_order）、处理过程记录、处理照片、材料消耗 |
| **下一环节** | 环节 6：验收归档 |
| **AI 服务调用** | 智能派单推荐：基于工单类型+人员技能+地理位置的最优分派 |
| **GIS 服务调用** | 路径规划：维修人员 → 问题点位的最优路径 |
| **数据存储** | `work_order` → PostgreSQL；处理照片 → OSS |

#### 环节 6：验收归档

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 工单状态变为"待验收" / 定时批量验收 |
| **输入数据** | 工单 ID、处理结果、验收标准、抽查照片 |
| **业务逻辑** | 1. 验收标准自动比对 → 2. 现场抽查（V2.0：AI图像对比） → 3. 验收结论 → 4. 归档并关闭工单 → 5. 触发报告生成 |
| **输出数据** | 验收记录（acceptance）、归档数据、报告草稿 |
| **下一环节** | 环节 7：分析优化 |
| **AI 服务调用** | V2.0图像对比：处理前后照片自动对比验证修复效果 |
| **GIS 服务调用** | 空间查询：同区域历史工单对比分析 |
| **数据存储** | `acceptance` → PostgreSQL；验收照片 → OSS |

#### 环节 7：分析优化

| 数据项 | 说明 |
|--------|------|
| **触发方式** | 定时任务（每日/每周） / 管理员手动触发 / 预警自动触发 |
| **输入数据** | 历史巡检数据、问题统计数据、工单处理数据、环境数据 |
| **业务逻辑** | 1. 多维度数据聚合 → 2. 趋势预测与异常检测 → 3. 热力图与空间分析 → 4. 生成优化建议 → 5. 更新巡检计划推荐参数 |
| **输出数据** | 分析报告、趋势预测、热力图、优化建议、计划调整方案 |
| **下一环节** | 反馈至环节 1（闭环） |
| **AI 服务调用** | 1. 预测分析：历史数据 → 趋势预测+异常检测；2. 智能优化：效果反馈 → 计划调整建议 |

### 5.6 数据安全规则

| 安全规则 | 具体措施 |
|---------|---------|
| **传输安全** | 全链路 HTTPS/WSS；内部服务间调用使用内网 HTTP（V1.0），V2.0升级为 mTLS |
| **存储安全** | 敏感字段加密存储（AES-256）；数据库访问最小权限原则 |
| **访问控制** | 基于 RBAC 的数据访问权限；行级数据隔离（不同园区数据不可互访） |
| **备份策略** | 每日全量备份 + 实时 binlog；OSS 跨区域冗余；备份数据保留 30 天 |
| **日志审计** | 所有数据写操作记录审计日志；敏感数据访问记录追踪 |

### 5.7 数据流设计约束

| 约束项 | 说明 |
|--------|------|
| **不做大数据平台** | V1.0 数据分析基于 PostgreSQL 聚合查询和基础报表，不引入 Hadoop/Spark |
| **实时性有限** | V1.0 实时数据仅包括位置更新和消息通知，不做实时流处理 |
| **离线优先** | 移动端支持离线数据采集和增量同步，保证数据不丢失 |
| **数据一致性** | 采用最终一致性模型，允许短暂不一致（如离线同步延迟），通过冲突解决策略保证数据完整性 |
| **存储成本控制** | 照片压缩存储、GPS 轨迹降采样、历史数据归档策略 |

### 6.7 部署约束与决策

| 约束项 | 决策 | 说明 |
|--------|------|------|
| **不做K8s/Docker** | V1.0采用传统虚拟机/物理机部署 | 降低运维复杂度，V2.0评估容器化收益后决策 |
| **不做微服务拆分** | V1.0单体应用，AI/GIS服务以独立进程部署在同一服务器 | 减少网络调用开销，V2.0按领域拆分微服务 |
| **单数据中心** | V1.0单机房部署，不做异地多活 | MVP阶段成本优先，V2.0评估灾备需求 |
| **AI服务优先第三方** | V1.0图像识别和NLP依赖阿里云等第三方云服务 | 降低算法团队投入，快速验证业务价值 |
| **GIS服务混合部署** | 地图渲染和路径规划用第三方API，空间分析用自研+PostGIS | 平衡成本和性能，核心空间计算自持 |
| **数据库主从架构** | PostgreSQL一主一从，满足V1.0读写需求 | 简单可靠，V2.0评估读写分离中间件 |

### 6.8 版本演进路线

#### V1.0 部署形态（当前）

- **应用层**：单体Java/Node.js应用，2节点负载均衡
- **AI层**：图像识别+NLP调用第三方云API，预测分析自研Python服务本地部署
- **GIS层**：地图渲染+路径规划调用高德API，空间分析自研服务+PostGIS
- **数据层**：PostgreSQL主从、Redis单节点/哨兵、OSS文件存储
- **运维**：传统虚拟机部署，手动发布，日志本地存储

#### V2.0 部署演进（规划）

| 演进项 | V1.0 | V2.0 |
|--------|------|------|
| **应用架构** | 单体应用 | 微服务拆分（计划/执行/工单/分析服务独立部署） |
| **容器化** | 虚拟机部署 | 评估Docker容器化，视团队能力决策K8s |
| **AI服务** | 第三方为主 | 自研模型服务化，引入模型版本管理和A/B测试 |
| **GIS服务** | 2D+第三方API | 3D BIM渲染自研，室内定位服务接入 |
| **数据层** | 关系型为主 | 引入时序数据库（InfluxDB/TDengine）、BIM模型存储 |
| **DevOps** | 手动发布 | CI/CD流水线，自动化测试，灰度发布 |
| **监控告警** | 基础日志 | 引入APM（应用性能监控）、链路追踪、业务指标监控 |
