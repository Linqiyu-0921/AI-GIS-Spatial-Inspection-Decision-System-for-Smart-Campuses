# 01 用户画像（Personas）

> 基于办公商务园区巡检场景构建的 5 个典型用户画像，用于指导功能设计优先级、交互细节及信息架构决策。
> ⚠️ 本文档所有痛点、目标、行为特征均标注 `[假设]`，因当前无真实用户调研数据支撑，后续需通过实地访谈、问卷及可用性测试验证。

---

## 画像一：巡检员（老手变体）

### 基础信息

| 维度 | 内容 |
|------|------|
| **姓名** | 张建国 `[假设]` |
| **年龄** | 48岁 `[假设]` |
| **职位** | 资深巡检员 `[假设]` |
| **从业经验** | 10年 `[假设]` |

### 角色描述

张建国在某商务园区从事巡检工作已满10年，对园区内3栋写字楼的每一个角落都了如指掌，闭着眼睛都能走完巡检路线。他熟悉所有设备的运行规律，能凭声音判断空调机组是否异常，凭气味察觉消防隐患。作为班组里的"老师傅"，他经常带新人，但自己依然需要每天完成固定的巡检任务。`[假设]`

### 核心痛点

1. **[假设]** 每天重复巡同样的路线，发现的问题越来越少，但制度要求必须按频次完成，感觉大量时间在"走形式"。
2. **[假设]** 发现问题后需要手写记录，回到办公室再录入系统，同样的信息填两遍，效率低下且容易遗漏细节。
3. **[假设]** 上报的问题经常"石沉大海"，不知道派给谁处理了、处理到哪一步了，缺乏闭环反馈让他觉得自己的工作价值不被看见。

### 核心目标

1. **[假设]** 减少无意义的重复劳动，把精力集中在真正有问题的地方。
2. **[假设]** 让自己发现的问题能被管理层看到、被责任方解决，形成有效闭环。
3. **[假设]** 学会使用数字化工具替代纸质工单，但学习成本不能太高，最好"一学就会"。

### 行为特征

1. **[假设]** 习惯按固定路线和时间节点巡检，身体记忆强于流程记忆，不喜欢被系统强制改变既有节奏。
2. **[假设]** 对新工具持保守态度，但一旦证明"能省事"就愿意接受，对"花里胡哨"的功能本能排斥。
3. **[假设]** 极度重视反馈闭环——希望看到自己上报的每一条问题都有回应、有结果，这是他工作成就感的主要来源。

### 典型使用场景

**场景A：日常巡检上报**
> 早上8点领取当日巡检任务 → 按固定路线逐点巡检 → 发现消防通道堆放杂物 → 拍照+语音快速上报 → 系统自动识别并创建工单 → 继续巡检 → 下班前查看工单处理状态，显示"已派单给保洁主管"。

**场景B：带教新人辅助**
> 新人李明辉跟随张建国学习 → 张建国演示如何在APP中完成上报 → 指导李明辉使用导航找到隐蔽点位 → 通过系统查看新人当日漏检项，针对性提醒。

### 技术熟练度

智能手机基础操作（微信、拍照、扫码），不熟悉复杂APP交互；长期使用纸质工单和Excel表格，对数据录入类操作有耐心但厌恶重复劳动。`[假设]` 首次使用新系统期望学习时间在5分钟以内。

## 画像二：巡检员（新手变体）

### 基础信息

| 维度 | 内容 |
|------|------|
| **姓名** | 李明辉 `[假设]` |
| **年龄** | 24岁 `[假设]` |
| **职位** | 巡检员 `[假设]` |
| **从业经验** | 入职3个月 `[假设]` |

### 角色描述

李明辉是刚入职3个月的新员工，目前跟随张建国等老员工学习巡检流程，逐步从辅助执行过渡到独立承担B区2栋写字楼的巡检任务。他对园区建筑结构、设备分布、风险点位还不够熟悉，处于"按清单办事"的阶段，缺乏基于经验的判断能力。`[假设]`

### 核心痛点

1. **[假设]** 不熟悉巡检路线和点位分布，容易漏检或走冤枉路，担心因漏检导致安全事故被追责。
2. **[假设]** 不知道哪些区域和设备属于高风险，需要重点关注，对"正常状态"和"异常状态"的区分标准模糊。
3. **[假设]** 发现问题后不确定严重程度——是该当场简单处理、上报维修、还是启动应急响应？判断依据不清晰。

### 核心目标

1. **[假设]** 在3个月内全面熟悉巡检路线和标准流程，做到独立上岗不遗漏、不误判。
2. **[假设]** 获得清晰的风险等级指引和判断标准，遇到问题时知道"该做什么、做到什么程度"。
3. **[假设]** 快速获得老员工的隐性经验（如"这个点位冬天容易出问题"），缩短学习曲线。

### 行为特征

1. **[假设]** 重度依赖导航和系统提示，对APP接受度极高，习惯被"引导式"操作，不喜欢需要自己判断的开放式界面。
2. **[假设]** 每次巡检都需要对照清单逐项确认，对"已完成/未完成"的视觉反馈非常敏感，需要明确的进度提示。
3. **[假设]** 遇到不确定的问题时倾向于拍照留存、询问老员工或系统辅助判断，不轻易自行决策。

### 典型使用场景

**场景A：新手引导式巡检**
> 打开APP → 查看今日巡检路线和地图导航 → 到达第一个点位 → 按系统checklist逐项检查（照明/消防器材/通道畅通）→ 逐项打勾确认 → 系统提示"该点位已检查完毕，下一站：B栋3层弱电间"。

**场景B：异常判断辅助**
> 巡检中发现空调外机异响 → 拍照上传 → 系统AI识别辅助判断"疑似风扇叶片松动，建议报修等级：中等" → 李明辉参考系统建议提交工单 → 工单自动关联历史类似案例供他学习。

### 技术熟练度

智能手机重度用户（短视频、外卖、社交软件、手游），对各类APP上手极快，喜欢交互流畅、视觉反馈丰富的界面。`[假设]` 对需要手动输入大量文字的场景不耐烦，偏好语音输入、拍照上传、一键确认等低摩擦操作。

## 画像三：物业经理

### 基础信息

| 维度 | 内容 |
|------|------|
| **姓名** | 王丽华 `[假设]` |
| **年龄** | 35岁 `[假设]` |
| **职位** | 物业经理 `[假设]` |
| **管理范围** | 3个园区，团队规模约20人 `[假设]` |

### 角色描述

王丽华管理着3个办公商务园区的物业运营工作，直接管辖约20人的巡检和维修团队。她处于"承上启下"的关键位置：向上要向园区负责人汇报运营绩效，向下要保障巡检执行到位、工单及时闭环。她的日常被会议、报表、突发状况切割成碎片，最缺的是"一眼看清全局"的工具。`[假设]`

### 核心痛点

1. **[假设]** 不知道今天的巡检是否按时完成、是否全覆盖，只能依赖巡检员口头汇报或事后抽查，存在信息盲区。
2. **[假设]** 工单经常积压，但缺乏有效的优先级判断和督办机制，不知道哪些最紧急、该催谁、催了有没有用。
3. **[假设]** 月底写运营报告需要手工汇总各园区数据，Excel公式复杂、容易出错，耗费大量时间且数据口径不统一。
4. **[假设]** 巡检员报上来的问题描述经常模糊（如"灯坏了"），需要反复电话沟通确认具体位置、设备型号、严重程度。

### 核心目标

1. **[假设]** 实时掌控3个园区的巡检执行状态和工单闭环进度，从"事后救火"转向"事前预防"。
2. **[假设]** 建立标准化的工单分级和督办机制，确保P0/P1级问题在SLA时限内100%响应。
3. **[假设]** 月度运营报告能够自动生成，数据准确、口径统一，可直接用于向上汇报。

### 行为特征

1. **[假设]** 每天早上到岗后第一件事情是花10-15分钟查看昨日巡检完成率和未闭环工单清单，形成固定习惯。
2. **[假设]** 对超时工单和重复出现问题的区域高度敏感，会立即介入督办或要求增加巡检频次。
3. **[假设]** 需要跨园区横向对比数据（如A园区vs B园区的故障率、SLA达成率），以识别管理短板。

### 典型使用场景

**场景A：晨间运营概览**
> 早上8:30登录PC后台 → 首页一屏查看3个园区昨日巡检完成率（A:98%, B:100%, C:95%）→ 发现C园区有2张P1工单超时 → 点击红色预警卡片 → 查看工单详情和责任人 → 一键发起督办通知。

**场景B：月度报告生成**
> 月底最后一个工作日 → 进入"运营报告"模块 → 选择统计周期（上月1日-31日）→ 系统自动生成包含巡检覆盖率、工单闭环率、平均响应时长、TOP5问题类别等指标的图文报告 → 预览后导出PDF → 直接发送给园区负责人。

### 技术熟练度

熟练使用Office三件套、钉钉/企业微信等办公协作工具，能接受并快速上手Web后台管理系统。`[假设]` 偏好数据可视化（图表、仪表盘）胜于纯表格，对需要多次点击才能到达的功能有耐心但会记录效率损耗。

## 画像四：技术主管

### 基础信息

| 维度 | 内容 |
|------|------|
| **姓名** | 陈志远 `[假设]` |
| **年龄** | 42岁 `[假设]` |
| **职位** | 技术主管 `[假设]` |
| **职责范围** | 设备维保计划制定、工单技术审核、重大故障应急指挥 `[假设]` |

### 角色描述

陈志远是园区设备运维的技术负责人，管理着电梯、空调、供电、给排水、消防等核心设备的维保体系。他需要从专业技术角度判断工单处置方案是否合理、设备故障是否具有趋势性、维保资源如何优化配置。他的核心价值在于把"被动维修"转变为"预防性维护"。`[假设]`

### 核心痛点

1. **[假设]** 设备故障呈现上升趋势，但缺乏结构化的历史数据和趋势分析能力，只能凭经验被动响应，无法提前预警。
2. **[假设]** 不清楚哪些区域、哪些类型的设备老化最严重，维保预算分配缺乏数据依据，经常"撒胡椒面"。
3. **[假设]** 工单处置质量参差不齐，不同维修人员的方案和验收标准不统一，缺乏标准化的技术审核依据。
4. **[假设]** 应急事件发生时，无法快速定位最近的可用维修人员、备用设备和关键备件库存位置。

### 核心目标

1. **[假设]** 建立设备故障的趋势预测能力，将80%的维修工作从"事后抢修"转变为"计划性维护"。
2. **[假设]** 基于设备年龄、故障频率、维修成本等数据，科学制定年度维保预算和更新改造计划。
3. **[假设]** 实现工单处置方案的标准化审核和验收，确保同一类问题有统一的技术标准。

### 行为特征

1. **[假设]** 高度关注设备故障的空间分布和时间规律，习惯通过热力图、趋势图寻找聚类特征。
2. **[假设]** 制定维保计划前会下钻查看历史工单数据，分析故障频率、根因分类、处置时长等维度。
3. **[假设]** 应急事件时需要分钟级的资源调度响应，对"地图上快速定位"的需求极其强烈。

### 典型使用场景

**场景A：设备问题分析**
> 周一早上登录后台 → 查看设备问题热力图 → 发现C区电梯在近30天内故障上报5次，形成明显聚类 → 下钻查看历史工单详情 → 分析故障类型（门机故障3次、控制系统2次）→ 判断该电梯门机系统老化 → 制定专项维保计划 → 更新设备台账并设置预警阈值。

**场景B：应急资源调度**
> 接到B栋2层空调机房漏水告警 → 打开应急指挥界面 → 地图显示最近维修人员位于B栋1层 → 查看其当前任务状态（空闲）→ 一键派单并附带位置导航 → 同时查看最近备件库库存（距离200米，有对应型号密封件）→ 同步通知值班经理。

### 技术熟练度

熟悉设备管理系统和工单系统，能理解技术参数和统计图表，对GIS地图操作有一定基础。`[假设]` 偏好专业性强的功能界面，对过于"傻瓜化"的操作感到束缚，但需要复杂分析功能时有清晰的操作指引。

## 画像五：园区负责人

### 基础信息

| 维度 | 内容 |
|------|------|
| **姓名** | 赵明远 `[假设]` |
| **年龄** | 50岁 `[假设]` |
| **职位** | 园区负责人 `[假设]` |
| **关注重点** | 成本控制、安全合规、运营绩效 `[假设]` |

### 角色描述

赵明远统筹整个园区的运营管理，是园区物业团队的最高决策者。他的核心关切是：园区是否安全合规、运营成本是否合理、租户满意度是否达标。他不直接操作任何业务系统，而是通过报表、简报和会议获取信息，基于数据做出战略级决策，并向上级公司或业主方汇报。`[假设]`

### 核心痛点

1. **[假设]** 缺乏全局视角的运营态势感知，当前依赖下属层层汇报，信息经过过滤后可能失真或滞后。
2. **[假设]** 重大决策（如是否更换某批次电梯、是否增加安保预算）缺乏量化数据支撑，主要靠经验和谈判。
3. **[假设]** 不知道园区安全风险的整体水平和变化趋势，对"哪里最危险、哪里最需要投入"没有清晰判断。
4. **[假设]** 向上级公司或业主方汇报时，缺乏直观、权威的量化数据和可视化材料，说服力不足。

### 核心目标

1. **[假设]** 基于数据而非仅凭经验做决策，每一笔预算投入都有对应的ROI或风险降低量化依据。
2. **[假设]** 建立园区运营的健康度指标体系，做到"看一眼就知道哪里正常、哪里异常"。
3. **[假设]** 向上汇报时拥有权威、实时、可视化的数据支撑，提升专业形象和谈判筹码。

### 行为特征

1. **[假设]** 不关注单条工单的具体内容，只关注趋势性指标和全局健康度，需要"从宏观到微观"的钻取能力。
2. **[假设]** 习惯在每周一早上查看上周运营简报，月底查看月度分析报告，形成固定节奏。
3. **[假设]** 遇到重大事件（如安全事故、政府检查）时，需要即时获取园区态势地图和关键指标快照。

### 典型使用场景

**场景A：周运营概览**
> 周一早上8:00收到系统自动生成的周报邮件 → 在iPad上打开 → 查看核心指标卡片：巡检完成率趋势（近4周）、问题热力图变化（本周vs上周）、SLA达成率、安全事件数量 → 发现D区问题频次连续两周上升 → 批注"请物业经理排查D区管理短板，本周五前汇报"。

**场景B：月度决策汇报**
> 月底准备向业主方汇报 → 打开系统决策看板 → 查看"月度运营健康度"评分（85分，环比+3分）→ 下钻查看扣分项（设备老化率偏高、2起轻微安全事件）→ 导出大屏适配的汇报PDF → 用于支撑下季度预算申请。

### 技术熟练度

日常使用iPad查看邮件和报表，不直接操作业务系统，所有操作通过秘书或下属代为执行。`[假设]` 偏好极简的信息呈现方式（红/黄/绿灯、评分卡、趋势箭头），对复杂界面和专业术语有耐心但要求有人解释。

## 设计原则提炼

基于以上画像，系统交互设计遵循以下优先级：

1. **移动端极简原则**：张建国和李明辉在户外移动场景使用，核心操作（上报、导航、确认）≤3步完成。
2. **反馈闭环原则**：所有上报必须有状态追踪，让执行者（巡检员）和管理者（物业经理）都看到进度。
3. **分层信息架构**：同一套数据按角色呈现不同粒度——底层工单数据 → 中层团队看板 → 高层决策简报。
4. **零学习成本原则**：新手上路需要引导式交互，老手需要快捷路径，管理层需要"一眼看懂"的可视化。
5. **假设验证清单**：后续用户研究需重点验证以上 `[假设]` 标注的痛点和目标是否符合真实用户场景。
