# 08 版本规划：V1.0 与 V2.0 路线图

> 明确当前版本（V1.0）的产品边界与未来版本（V2.0）的演进方向，为开发排期、资源投入和面试展示提供清晰的阶段划分依据。

---

## 一、版本总览

| 维度 | V1.0（当前） | V2.0（未来） |
|------|-------------|-------------|
| **目标** | MVP 落地，验证核心场景闭环 | 规模化扩展，构建平台级产品 |
| **园区范围** | 单园区 | 多租户、多园区 |
| **场景** | 3 个（日常巡检/设备检查/应急事件） | 3 个核心 + 扩展场景（能耗/停车/安防） |
| **角色** | 2 个（巡检员 + 管理层） | 2 个核心 + 扩展角色（租户管理员/外部访客） |
| **AI 能力** | 3 个（图像识别/NLP 结构化/预测分析） | 3 个核心 + 扩展（语音交互/知识图谱/数字孪生推理） |
| **GIS 能力** | 3 个（空间可视化/问题聚类/缓冲区分析） | 3 个核心 + 扩展（3D 漫游/BIM 联动/实时仿真） |
| **地图** | 2D 园区地图 | 3D 园区/BIM 集成 + 数字孪生底座 |
| **部署** | 单体部署 | 微服务 + 多租户隔离 |

## 二、V1.0 详细边界

### 2.1 V1.0 包含的内容

| 模块 | 具体内容 |
|------|---------|
| **场景闭环** | 日常巡检（8 步闭环）、设备检查（7 步闭环）、应急事件（7 步并行响应闭环） |
| **用户角色** | 巡检员（移动端 APP）、管理层（PC 后台，含物业经理/技术主管） |
| **AI 能力** | 图像识别（缺陷分类/外观识别）、NLP 结构化（语音转工单/文本提取）、预测分析（巡检频次推荐/设备劣化预测） |
| **GIS 能力** | 空间可视化（2D 园区地图/问题标注）、问题聚类（DBSCAN/OPTICS 热力图）、缓冲区分析（应急影响半径） |
| **工单系统** | 4 步工单（创建→分配→处置→验收），含 SLA 时限与升级机制 |
| **数据闭环** | 分析引擎驱动计划自优化（频次/路线/点位/人员调整） |
| **移动端** | iOS + Android 双端，离线缓存 + 断网续传 |
| **PC 后台** | 计划编排、工单管理、数据分析、人员管理、设备台账 |

### 2.2 V1.0 明确不包含的内容

> 以下功能在 V1.0 阶段**明确不做**，避免范围蔓延。

| 不包含项 | 原因 | 计划版本 |
|---------|------|---------|
| **多租户/多园区** | 单园区 MVP 足够验证核心闭环；多租户涉及数据隔离、权限重构，复杂度高 | V2.0 |
| **3D 园区地图** | 3D 建模成本高，2D 地图已满足 V1.0 空间可视化需求 | V2.0 |
| **数字孪生** | 实时仿真、IoT 全量接入、虚实映射技术栈重，V1.0 以工单闭环为核心 | V2.0 |
| **BIM 深度集成** | 设备检查场景使用简化 BIM 楼层定位，不做全量 BIM 模型加载和属性联动 | V2.0 |
| **能耗管理** | 属于扩展场景，与巡检主线关联弱 | V2.0 |
| **停车管理** | 属于扩展场景，需独立硬件接入 | V2.0 |
| **视频监控全量接入** | 应急场景仅做视频联动核查，不做 7×24 视频 AI 巡检 | V2.0 |
| **外部系统对接（ERP/财务）** | 工单闭环无需财务结算，接口对接增加集成复杂度 | V2.0 |
| **多语言/国际化** | 目标用户为国内园区，单一中文足够 | 待定 |
| **小程序/H5 轻量端** | 巡检员高频使用场景，原生 APP 体验更优 | 待定 |

## 三、V1.0 时间线建议

### 3.1 阶段划分（总时长：约 3 个月）

| 阶段 | 周期 | 核心目标 | 交付物 |
|------|------|---------|--------|
| **Phase 1：基础搭建** | 第 1-2 周 | 技术选型、架构设计、数据库设计、开发环境搭建 | 技术方案文档、数据库 ER 图、API 接口定义 |
| **Phase 2：核心闭环** | 第 3-6 周 | 日常巡检全流程（计划→执行→发现→工单→处置→验收→分析） | 可演示的日常巡检闭环 |
| **Phase 3：场景扩展** | 第 7-9 周 | 设备检查 + 应急事件闭环 | 三个场景全部可演示 |
| **Phase 4：AI/GIS 嵌入** | 第 10-11 周 | 图像识别、NLP、预测分析、GIS 可视化/聚类/缓冲区 | AI 辅助上报、GIS 热力图、SLA 升级机制 |
| **Phase 5：集成测试** | 第 12 周 | 端到端测试、性能调优、Bug 修复、文档补全 | V1.0 可交付版本 |

### 3.2 里程碑

```
Week 2  [M1] 技术方案评审通过
Week 6  [M2] 日常巡检闭环演示
Week 9  [M3] 三场景全量演示
Week 11 [M4] AI/GIS 能力集成演示
Week 12 [M5] V1.0 正式交付
```

### 3.3 资源假设

- 开发团队：前端 2 人 + 后端 2 人 + AI 算法 1 人 + 产品 1 人
- 设计资源：UI 设计 1 人（兼职，前期介入）
- 硬件资源：测试用园区 1 个、测试手机 2-3 部、服务器 1 台

> [假设] 上述时间线基于标准敏捷开发节奏，实际排期需根据团队规模和外部依赖调整。

## 四、V2.0 演进方向

### 4.1 V2.0 核心升级点

| 升级维度 | V1.0 现状 | V2.0 目标 |
|---------|----------|----------|
| **多租户架构** | 单园区，数据无隔离 | 支持多园区独立部署/ SaaS 化多租户，数据完全隔离 |
| **3D 园区/BIM** | 2D 地图 + 简化 BIM 楼层定位 | 3D 园区可视化，BIM 模型全量加载，设备属性联动 |
| **扩展场景** | 仅巡检相关 3 场景 | 新增能耗监控、停车管理、安防联动等扩展场景 |
| **数字孪生底座** | 无 | 构建园区数字孪生底座，实现实时数据映射、仿真预测 |
| **AI 能力扩展** | 3 个核心能力 | 语音交互（巡检员语音操控）、知识图谱（设备关联推理）、数字孪生推理 |
| **GIS 能力扩展** | 3 个核心能力 | 3D 漫游、BIM-GIS 联动、实时仿真（人流/车流模拟） |
| **部署架构** | 单体部署 | 微服务架构，支持弹性扩缩容 |
| **开放平台** | 无 | 开放 API 接口，支持第三方系统对接 |

### 4.2 V2.0 时间线建议（总时长：6-12 个月）

| 阶段 | 周期 | 核心目标 |
|------|------|---------|
| **Phase 1：架构重构** | 第 1-2 个月 | 单体拆微服务、多租户数据隔离、权限体系重构 |
| **Phase 2：3D/BIM** | 第 2-4 个月 | 3D 园区建模、BIM 引擎集成、3D 可视化底座 |
| **Phase 3：场景扩展** | 第 4-6 个月 | 能耗/停车/安防场景接入，IoT 全量接入 |
| **Phase 4：数字孪生** | 第 6-9 个月 | 实时数据映射、仿真引擎、预测性维护 |
| **Phase 5：平台化** | 第 9-12 个月 | 开放 API、SaaS 化运营、多园区标杆客户落地 |

> [假设] V2.0 时间线跨度较大，需根据 V1.0 市场反馈和技术验证结果动态调整优先级。

## 五、版本决策依据

### 5.1 为什么 V1.0 不做多租户？

- **验证优先**：单园区的完整闭环已足够验证产品价值（SLA 驱动、AI 辅助、GIS 可视化）
- **复杂度控制**：多租户涉及数据库隔离、权限重构、配置化管理，会将 V1.0 工期拉长 50% 以上
- **面试展示**：PM 作品集的核心是「产品设计完整性」，而非「技术架构先进性」；单园区深度闭环比多园区浅层覆盖更有说服力

### 5.2 为什么 V1.0 不做 3D/数字孪生？

- **成本权衡**：3D 建模和数字孪生底座需额外 2-3 个月，且需专业 3D 美术和引擎开发资源
- **需求匹配**：V1.0 目标用户（园区物业）的核心痛点是「问题闭环慢、巡检效率低」，2D 地图 + SLA 驱动已能显著改善
- **演进路径**：V1.0 的 2D GIS 数据和 BIM 楼层定位数据可为 V2.0 的 3D 底座复用，不阻塞未来扩展

### 5.3 为什么 V2.0 才做扩展场景？

- **主线聚焦**：能耗、停车与巡检主线关联弱，独立场景需独立硬件接入和数据源
- **资源聚焦**：V1.0 团队规模有限，集中资源打磨 3 个核心场景的闭环体验
- **市场验证**：V1.0 落地后收集实际使用数据，再决定扩展场景的优先级排序

## 六、版本对照总表

| 维度 | V1.0 | V2.0 |
|------|------|------|
| 园区范围 | 单园区 | 多租户/多园区 |
| 地图 | 2D | 3D/BIM |
| 场景 | 日常巡检、设备检查、应急事件 | + 能耗、停车、安防 |
| AI | 图像识别、NLP、预测分析 | + 语音交互、知识图谱、数字孪生推理 |
| GIS | 可视化、聚类、缓冲区 | + 3D 漫游、BIM 联动、实时仿真 |
| 部署 | 单体 | 微服务 |
| 目标 | MVP 验证 | 平台规模化 |
| 工期 | ~3 个月 | ~6-12 个月 |

---

> **引用来源**：版本规划基于竞品分析中「海康/广联达产品功能冗余、落地周期长」的洞察，以「小步快跑、闭环优先」为设计原则，与 V1.0 不做 3D/多租户的约束保持一致。
