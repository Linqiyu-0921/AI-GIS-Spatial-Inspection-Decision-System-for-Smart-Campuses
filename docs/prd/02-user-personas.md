# 02 用户画像

> 基于办公商务园区场景构建5个典型用户画像，指导功能设计优先级和交互细节。
> ⚠️ 所有画像均标注 `[假设]` 前缀，因当前无真实用户调研数据，后续需通过实地访谈验证。

---

## 画像一：巡检员（老手变体）

> `[假设]` 张建国 — "闭着眼都能走完的路线，但我缺一个好用的工具"

| 维度 | 内容 |
|------|------|
| **姓名** | 张建国 `[假设]` |
| **年龄** | 42岁 `[假设]` |
| **角色** | 资深巡检员，从业8年 `[假设]` |
| **日常职责** | 负责A区3栋写字楼的日常巡检，包含公共区域、设备机房、消防通道 |
| **技术熟练度** | 智能手机基础操作（微信、拍照），不熟悉复杂APP；习惯纸质工单 |
| **核心痛点** | ①每天重复巡同样的路线，发现的问题越来越少但不能不巡②发现问题后要手写记录再回办公室录入系统，效率低③上报的问题经常"石沉大海"，不知道处理到哪一步了 |
| **核心目标** | 减少无意义的重复劳动，让自己发现的问题能被看到、被解决 |
| **行为特征** | 习惯按固定路线巡检；对新工具抵触但愿意学"能省事"的功能；重视反馈——希望自己上报的问题有回应 |
| **典型使用场景** | 早上8点领取当日巡检任务 → 按路线逐点巡检 → 发现消防通道堆放杂物 → 拍照+语音上报 → 系统自动识别并创建工单 → 继续巡检 → 下班前查看工单处理状态 |

**设计启示：**
- 移动端操作必须极简：拍照→语音→提交，≤3步完成上报
- 需要工单状态反馈：让巡检员看到"我报的问题处理到哪了"
- 降低学习成本：首次使用引导 ≤5min

## 画像二：巡检员（新手变体）

> `[假设]` 李明辉 — "刚入职两周，路线还不熟，最怕漏检"

| 维度 | 内容 |
|------|------|
| **姓名** | 李明辉 `[假设]` |
| **年龄** | 25岁 `[假设]` |
| **角色** | 新入职巡检员，从业2个月 `[假设]` |
| **日常职责** | 跟随老员工学习巡检流程，逐步独立承担B区巡检任务 |
| **技术熟练度** | 智能手机重度用户（短视频、外卖、游戏），APP上手快 |
| **核心痛点** | ①不熟悉巡检路线，容易漏检点位②不知道哪些区域是高风险区域需要重点关注③发现问题不知道严重程度如何判断（该当场处理还是上报？） |
| **核心目标** | 快速上手巡检流程，不漏检、不误判 |
| **行为特征** | 依赖导航和提示；对APP接受度高；需要明确的操作指引和判断标准 |
| **典型使用场景** | 打开APP → 查看今日巡检路线和导航 → 到达点位后按checklist逐项检查 → 发现空调外机异常 → 系统辅助判断严重程度 → 按指引上报 → 系统提示"该点位已检查完毕，下一站：XX" |

**设计启示：**
- 巡检路线导航：类似地图导航的"下一站"引导
- Checklist模板化：每个点位的标准检查项明确列出
- AI辅助判断：图像识别帮助新手判断问题严重程度
- 进度可视化："已巡/待巡"实时显示，防止漏检

## 画像三：物业经理

> `[假设]` 王丽华 — "我需要知道今天有多少问题没处理完，为什么没处理完"

| 维度 | 内容 |
|------|------|
| **姓名** | 王丽华 `[假设]` |
| **年龄** | 36岁 `[假设]` |
| **角色** | 物业经理，管辖20人巡检团队 `[假设]` |
| **日常职责** | 巡检团队管理、工单分配与督办、租户投诉处理、月度运营报告 |
| **技术熟练度** | 熟练使用Office/钉钉/企业微信；能接受Web后台系统 |
| **核心痛点** | ①不知道今天的巡检是否按时完成、是否全覆盖②工单积压但不知道哪些最紧急、该催谁③月底写报告要手工汇总数据，耗时且容易出错④巡检员报上来的问题描述模糊，需要反复沟通确认 |
| **核心目标** | 实时掌控巡检执行状态和工单闭环进度，减少"救火式"管理 |
| **行为特征** | 每天早上花15分钟看巡检完成情况；关注超时未完成的工单；月底需要自动生成的运营报告 |
| **典型使用场景** | 早上登录PC后台 → 查看昨日巡检完成率和未闭环工单 → 发现有3张P1工单超时 → 系统自动升级并通知 → 督办处置 → 下午查看处置进度 → 下班前确认当日闭环率达标 |

**设计启示：**
- PC后台首页：巡检完成率、工单状态分布、SLA达成率一屏概览
- 异常高亮：超时工单红色标注，支持一键督办
- 自动报告：月度运营报告自动生成，可导出

## 画像四：技术主管

> `[假设]` 陈志远 — "设备故障率在上升，但我找不到规律"

| 维度 | 内容 |
|------|------|
| **姓名** | 陈志远 `[假设]` |
| **年龄** | 40岁 `[假设]` |
| **角色** | 技术主管，负责设备维保和工单技术审核 `[假设]` |
| **日常职责** | 设备维保计划制定、工单技术方案审核、重大故障应急指挥、设备台账管理 |
| **技术熟练度** | 熟悉设备管理系统；能理解技术数据和图表 |
| **核心痛点** | ①设备故障频发但缺乏趋势数据，只能被动响应②不知道哪些区域的设备老化最严重③工单处置质量参差不齐，缺乏标准化验收依据④应急事件响应时，不知道最近的可用资源在哪 |
| **核心目标** | 从被动维修转向预防性维护，提升设备可靠性 |
| **行为特征** | 关注设备故障趋势和空间分布；需要技术维度的数据分析；应急事件时需要快速定位资源 |
| **典型使用场景** | 登录后台 → 查看设备问题热力图 → 发现C区电梯故障聚类 → 下钻查看历史工单 → 分析故障频率和类型 → 制定针对性维保计划 → 审核重大工单技术方案 |

**设计启示：**
- GIS热力图：设备问题空间聚类，支持按设备类型筛选
- 趋势分析：故障频率时间序列，支持自定义时间范围
- 资源地图：应急时快速定位最近可用人员和备件

## 画像五：园区负责人

> `[假设]` 赵明远 — "我要看的是全局态势，不是具体工单"

| 维度 | 内容 |
|------|------|
| **姓名** | 赵明远 `[假设]` |
| **年龄** | 48岁 `[假设]` |
| **角色** | 园区负责人，统筹园区运营管理 `[假设]` |
| **日常职责** | 园区战略规划、重大决策审批、对外关系协调、运营绩效把控 |
| **技术熟练度** | 日常使用iPad查看报表；不直接操作系统，通过下属获取信息 |
| **核心痛点** | ①缺乏全局视角的运营态势感知②决策依赖下属汇报，缺乏数据支撑③不知道园区安全风险的整体水平和变化趋势④向上汇报时缺乏量化数据 |
| **核心目标** | 基于数据做决策，向上汇报有据可依 |
| **行为特征** | 不关注单条工单，关注趋势和全局指标；每周看一次运营简报；重大事件时需要态势地图 |
| **典型使用场景** | 周一早上收到系统自动生成的周报 → 查看巡检完成率趋势、问题热力图变化、SLA达成率 → 发现某区域问题频发 → 下发指令要求重点巡查 → 月底收到月度报告用于向上汇报 |

**设计启示：**
- 决策看板：大屏适配，核心指标一屏展示
- 自动周报/月报：系统生成，支持导出PDF
- 趋势对比：环比/同比数据，支持钻取下探
