## 一、背景与目标

### 1.1 市场背景

[假设] 中国智慧园区市场规模预计 2026 年突破 3000 亿元，年复合增长率约 15%。其中，园区运维管理（含巡检、安防、设备管理）占整体投入的 20%-25%，是仅次于基础设施建设的第二大细分领域。

办公商务园区（写字楼集群、CBD）区别于产业园区和制造业园区，具备以下特征：

| 特征 | 说明 |
|------|------|
| 空间形态 | 多栋高层/超高层写字楼 + 公共区域（广场、停车场、绿化带） |
| 管理对象 | 电梯、空调、消防、照明、门禁等楼宇设备为主 |
| 人员密度 | 高密度白领租户，对环境品质和安全敏感度高 |
| 物业模式 | 第三方物业托管为主，管理标准化程度参差不齐 |

### 1.2 办公园区巡检痛点

| 痛点 | 现状 | 后果 |
|------|------|------|
| **巡检靠腿、记录靠纸** | 巡检员手持纸质工单逐点打卡，发现问题手写记录 | 信息滞后、描述模糊、无法追溯 |
| **发现问题 ≠ 解决问题** | 巡检记录与工单系统割裂，发现隐患后需人工转录、手动派单 | 问题遗漏、响应延迟、责任不清 |
| **巡了什么 ≠ 该巡什么** | 巡检路线和频率固定，不根据历史数据动态调整 | 高风险区域巡检不足，低风险区域重复浪费 |
| **数据沉睡** | 巡检数据仅用于存档，不做趋势分析和决策支持 | 管理层无法预判风险，决策依赖经验 |
| **竞品方案不适配** | 海康偏安防硬件、广联达偏施工场景、开源方案无 AI/GIS | 办公园区缺乏针对性解决方案 |

### 1.3 产品愿景

**一句话定位：**

> 让每一次巡检都有据可依、每一个问题都闭环解决、每一条数据都驱动决策——从"记录巡了什么"升级为"决策该怎么巡"。

**差异化定位：**

本项目不做"又一个功能更全的园区管理系统"，而是聚焦于以 GIS 空间决策为核心引擎、以 AI 全流程嵌入为技术底座、以自优化闭环为进化机制的办公商务园区巡检决策系统。

### 1.4 核心目标

| 编号 | 目标 | 量化指标 | 衡量方式 |
|------|------|---------|---------|
| G1 | 问题发现到工单创建的时间缩短 | 从人工转录的 [假设] 30min 缩短至 ≤2min（AI 自动结构化 + 自动触发） | 工单创建时间戳 - 问题上报时间戳的均值 |
| G2 | 工单闭环率提升 | 从 [假设] 70% 提升至 ≥95%（SLA 驱动 + 超时升级） | 验收通过的工单数 / 总工单数 |
| G3 | 巡检覆盖率提升 | 从 [假设] 60% 提升至 ≥90%（覆盖盲区检测 + 路线优化） | GIS 覆盖分析：已巡点位 / 应巡点位 |
| G4 | 管理决策有数据支撑 | 100% 的巡检数据进入分析引擎，产出热力图、趋势报告 | 数据入库率 + 报告产出频率 |
| G5 | 系统易用性达标 | 巡检员端上手培训时间 ≤30min，管理层端核心操作 ≤3 步 | 新用户任务完成率测试 |

### 1.5 目标用户

| 角色 | 使用端 | 核心诉求 |
|------|--------|---------|
| **巡检员** | 移动端（APP/H5） | 快速接收任务、高效上报问题、减少重复劳动 |
| **管理层** | PC 后台（Web） | 掌控全局态势、追踪问题闭环、基于数据做决策 |

管理层进一步细分为：物业经理（日常运营）、技术主管（设备与工单）、园区负责人（战略决策）。

### 1.6 V1.0 产品范围

**包含：**

| 模块 | 范围 |
|------|------|
| 巡检任务管理 | 计划编排、任务下发、执行记录、完成确认 |
| 问题上报与工单 | 移动端上报（拍照 + 语音 + 文字）→ AI 结构化 → 自动/手动创建工单 → 分派 → 处置 → 验收 |
| AI 辅助识别 | 巡检现场图像识别（缺陷/隐患分类）、上报内容 NLP 结构化 |
| GIS 空间可视化 | 园区地图底图、巡检轨迹、问题点位标注、热力图聚类 |
| SLA 与升级 | 工单分级（P0-P3）、时限配置、超时自动升级与转派 |
| 数据看板 | 巡检完成率、工单闭环率、SLA 达成率、问题热力图 |
| 移动巡检端 | 巡检员登录、任务列表、地图导航、问题上报、工单处置 |

**不包含（V1.0 明确排除）：**

| 排除项 | 理由 |
|--------|------|
| 3D 数字孪生/BIM | V1.0 仅 2D GIS，3D 为后续版本 |
| IoT 设备对接 | V1.0 聚焦人工巡检，IoT 联动为 V2.0 |
| 多园区管理 | V1.0 单园区验证，多园区横向扩展为后续版本 |
| 财务/合同管理 | 超出巡检决策系统边界 |
| 无人机/机器人巡检 | V1.0 仅移动端人工巡检 |
| 第三方系统集成 | V1.0 独立运行，开放 API 为 V2.0 |

### 1.7 成功标准

V1.0 上线后 [假设] 3 个月内达成以下标准视为项目成功：

1. 至少 1 个办公商务园区完成试点部署
2. 巡检员端日活使用率 ≥80%
3. 工单平均闭环时间缩短 ≥50%（对比上线前基线）
4. 管理层周报/月报 100% 由系统数据自动生成
5. 用户满意度评分 ≥4.0/5.0（巡检员 + 管理层分别评估）

## 三、场景概述

### 3.1 场景一句话定义

| 场景 | 定义 | 频率 | 触发方式 |
|------|------|------|---------|
| **日常巡检** | 巡检员按计划路线对园区公共区域进行例行巡查，发现并上报环境、安全隐患 | 每日 1-2 次 | 计划任务自动下发 |
| **设备检查** | 对电梯、空调、消防、照明等关键设备进行专项检查，记录运行状态 | 每周/每月（按设备类型） | 计划任务 + 设备告警触发 |
| **应急事件** | 火灾、水管爆裂、电梯困人等突发事件的快速响应与处置 | 不定期 | 人工上报 / IoT 告警 / 系统检测 |

### 3.2 七步闭环流程骨架

所有巡检场景共享统一的七步闭环流程：

```
计划 → 执行 → 发现 → 工单 → 处置 → 验收 → 分析
```

| 步骤 | 名称 | 角色 | AI 嵌入 | GIS 嵌入 |
|------|------|------|--------|---------|
| **① 计划** | 巡检计划编排与下发 | 管理层编排，巡检员接收 | 基于历史数据推荐巡检频率和路线（自优化） | 覆盖分析：检测巡检盲区，确保无遗漏 |
| **② 执行** | 现场巡检执行 | 巡检员按路线逐点巡检 | 巡检进度实时同步，异常区域预警 | 轨迹记录：GPS 定位 + 路线回放，确保真实到达 |
| **③ 发现** | 问题发现与上报 | 巡检员拍照/语音/文字上报 | 图像识别自动分类缺陷类型；NLP 结构化上报内容 | 定位打标：问题点位自动标注到 GIS 地图 |
| **④ 工单** | 工单创建与分派 | 系统自动创建，管理层分派 | 智能推荐处置人（基于技能 + 位置 + 负载） | 就近匹配：GIS 计算最近可用处置人员 |
| **⑤ 处置** | 问题处置与反馈 | 处置人（巡检员/外包/维修队） | 处置方案知识库推荐 | 导航指引：GIS 导航至问题点位 |
| **⑥ 验收** | 处置结果验收 | 管理层或巡检员验收 | 图像比对：处置前后照片 AI 对比验证 | 现场确认：定位验证是否在问题点位完成 |
| **⑦ 分析** | 数据分析与决策优化 | 管理层查看，系统自动产出 | 趋势预测、异常检测、优化建议 | 热力图聚类：空间维度的问题分布分析 |

### 3.3 闭环分支逻辑

| 分支 | 适用场景 | 流程 | 说明 |
|------|---------|------|------|
| **轻微当场闭环** | 地面有纸屑、灯泡需更换等 | 发现 → 当场处置 → 验收 → 分析 | 巡检员自行处理，拍照留证即可 |
| **严重走工单** | 设备故障、安全隐患、结构损坏等 | 发现 → 工单 → 分派 → 处置 → 验收 → 分析 | 需要专业人员处理，走完整工单流程 |

判断标准由 AI 辅助：图像识别问题类型 → 匹配严重等级 → 引导巡检员选择分支。

### 3.4 场景与闭环映射

**日常巡检：**

| 闭环步骤 | 具体表现 |
|---------|---------|
| 计划 | 管理层设定巡检路线、频率、点位；系统根据历史数据建议优化 |
| 执行 | 巡检员按"导航式"路线逐点巡检，每到一个点位完成 checklist |
| 发现 | 拍照 → AI 识别问题类型 → 语音补充描述 → NLP 结构化为工单字段 |
| 工单 | 系统自动创建工单 → AI 推荐最近可用人员 → 管理层确认分派 |
| 处置 | 处置人收到工单 → 导航至问题点位 → 处置 → 拍照上传 |
| 验收 | 巡检员下次巡检时复核 → 或管理层远程验收处置照片 |
| 分析 | 月度分析：识别问题高发区域 → 建议增加巡检频次或增设监控 |

**设备检查：**

| 闭环步骤 | 具体表现 |
|---------|---------|
| 计划 | 按设备维保规范设定检查周期（电梯月检、消防季检、空调半年检）；系统自动排期 |
| 执行 | 技术员到达设备点位 → 按设备 checklist 逐项检查（运行参数、外观、异常声音） |
| 发现 | 录音 + 拍照 → AI 识别异常 → 自动提升严重等级 |
| 工单 | 系统创建工单 → 自动通知技术主管 → AI 推荐维保单位 |
| 处置 | 维保单位接单 → 现场检修 → 更换零部件 → 上传维修报告 |
| 验收 | 技术主管审核维修报告 → 安排复检 → 确认设备恢复正常 |
| 分析 | 趋势分析：识别故障率上升设备 → 建议大修或更换 |

**应急事件：**

| 闭环步骤 | 具体表现 |
|---------|---------|
| 计划 | 预设应急预案（火灾、水管爆裂、电梯困人等）；系统自动匹配响应流程 |
| 执行 | 事件触发 → 系统自动识别事件类型 → 推送应急预案 → 通知相关人员 |
| 发现 | 紧急上报 → AI 识别事件类型 → 自动触发应急流程 |
| 工单 | 创建 P0 应急工单 → 同时通知物业经理 + 维修队 + 安保 → GIS 定位最近可用人员 |
| 处置 | 多线程并行：维修队处置 + 安保疏散 + 管理层协调资源 |
| 验收 | 现场恢复 → 管理层确认事件关闭 |
| 分析 | 事件复盘：分析响应瓶颈 → 优化应急预案 |

### 3.5 应急响应分级与 SLA

| 级别 | 定义 | 响应时限 | 示例 | 升级规则 |
|------|------|---------|------|---------|
| **P0** | 重大安全事件，需立即响应 | ≤15min | 火灾、电梯困人、水管爆裂 | 15min 未响应自动升级至园区负责人 |
| **P1** | 严重隐患，需尽快处理 | ≤2h | 消防通道堵塞、设备重大故障 | 2h 未处理自动升级至技术主管 |
| **P2** | 一般问题，按计划处理 | ≤24h | 照明损坏、墙面污损 | 24h 未处理自动升级至物业经理 |
| **P3** | 轻微问题，可批量处理 | ≤72h | 标识牌脱落、绿化缺失 | 72h 未处理提醒物业经理 |

### 3.6 AI/GIS 嵌入总览

**AI 嵌入点：**

| 环节 | AI 能力 | 具体作用 |
|------|--------|---------|
| 计划 | 预测分析 | 基于历史数据推荐巡检频率和路线优化 |
| 执行 | 实时预警 | 异常区域实时提醒（如连续多日未巡检的点位） |
| 发现 | 图像识别 + NLP | 自动识别缺陷类型 + 结构化上报内容 |
| 工单 | 智能推荐 | 基于技能、位置、负载推荐最佳处置人 |
| 处置 | 知识库推荐 | 匹配历史类似问题的处置方案 |
| 验收 | 图像比对 | 处置前后照片 AI 对比，自动判断是否修复 |
| 分析 | 趋势预测 + 异常检测 | 预测风险区域、识别异常模式 |

**GIS 嵌入点：**

| 环节 | GIS 能力 | 具体作用 |
|------|---------|---------|
| 计划 | 覆盖分析 | 检测巡检盲区，确保路线全覆盖 |
| 执行 | 轨迹记录 | GPS 定位 + 路线回放，验证巡检真实性 |
| 发现 | 定位打标 | 问题点位自动标注到园区地图 |
| 工单 | 就近匹配 | 空间计算最近可用处置人员 |
| 处置 | 导航指引 | GIS 导航至问题点位 |
| 验收 | 现场确认 | 定位验证处置是否在正确点位完成 |
| 分析 | 热力图聚类 | DBSCAN/OPTICS 聚类识别问题热区 |

### 3.7 场景间关联

三个场景并非孤立，而是共享数据、相互驱动：

- **日常巡检发现设备异常** → 自动触发设备检查场景
- **设备检查发现重大隐患** → 自动触发应急事件流程
- **应急事件处置完成后** → 数据进入分析引擎 → 优化日常巡检计划
- **分析引擎发现趋势** → 反向调整三个场景的计划（自优化闭环）

## 四、场景①：日常巡检全流程

日常巡检从计划编排到数据回流共 8 步，形成完整闭环：

```
①计划 → ②执行 → ③发现 → ④分支判断 → ⑤工单/当场处置 → ⑥处置 → ⑦验收 → ⑧分析
                                                                  ↑              |
                                                                  |___ 数据回流自优化 __|
```

> 分支判断（④）是七步骨架中"发现"与"工单"之间的新增步骤，由 AI 辅助完成严重程度判断，决定走轻微当场闭环还是严重工单流程。

### ① 计划 — 巡检计划编排与下发

| 维度 | 内容 |
|------|------|
| **角色** | 管理层（物业经理 王丽华）编排 → 巡检员（张建国/李明辉）接收 |
| **输入** | 历史巡检数据、园区布局图、问题热力图、上期分析报告 |
| **输出** | 当日巡检计划（含路线、点位、checklist、预计时长） |
| **触发方式** | 系统每日自动下发（晨会前完成），管理层可手动调整 |

**流程描述：**

1. 系统于每日 06:00 自动生成当日巡检计划，分配至对应巡检员的移动端
2. 巡检员打开 APP 后查看今日任务：巡检路线、点位清单、各点位 checklist、预计完成时间

### 数据回流闭环

分析（⑧）产出的优化建议，经管理层确认后回流至计划（①），驱动巡检计划自优化：

```
⑧分析 → 优化建议 → 管理层确认 → ①计划自动更新 → ②执行调整后的巡检
```

| 调整类型 | 触发条件 | 示例 |
|---------|---------|------|
| **频次调整** | 某区域连续 N 天无问题（降频）或问题频发（升频） | A 区连续 30 天无问题→降频至每周 2 次；B 区 7 天内 5 次堆物→升频至每日 2 次 |
| **路线调整** | 热力图发现新增问题高发区或覆盖盲区 | GIS 聚类发现 D 区新增隐患→将 D 区纳入巡检路线 |
| **点位调整** | 某点位长期无问题（移除）或新发现问题区域（新增） | E 区标识牌点位连续 60 天无异常→移除；F 区绿化带新发现问题→新增点位 |
| **人员调整** | 巡检员漏检率高或巡检效率低 | 李明辉连续 3 天漏检率 >10%→安排老员工张建国带教 1 周 |

## 五、场景②：设备专项检查全流程

设备专项检查从计划编排到数据分析共 7 步，形成完整闭环：

```
①计划 → ②执行 → ③发现 → ④工单 → ⑤处置 → ⑥技术验收 → ⑦分析
                                                    ↑              |
                                                    |__ 数据回流自优化 __|
```

### 与日常巡检的核心差异

| 维度 | 日常巡检 | 设备专项检查 |
|------|---------|-------------|
| **驱动方式** | 路线驱动（按地理路线逐点巡检） | 检查清单驱动（按设备 checklist 逐项检查） |
| **定位方式** | GPS/GIS 室外定位 | BIM/室内定位（设备在楼层内部） |
| **AI 能力** | 通用图像识别 + NLP 结构化 | 外观识别（锈蚀/泄漏/变形）+ OCR 表盘读数 |
| **验收方式** | 巡检员复核或管理层远程验收照片 | 技术主管专业验收 + 功能测试 |
| **频率** | 每日 1-2 次 | 按设备类型（周检/月检/季检/半年检） |
| **处置复杂度** | 简单维修（保洁/更换灯泡） | 专业维保单位（电梯/消防/电力） |
| **闭环分支** | 轻微当场闭环/严重走工单 | 全部走工单（设备问题无轻微之说） |

### 设备检查清单模板（摘要）

**电梯检查清单（月检，8 项）：** 运行平稳性、门机系统、安全装置、通讯系统、照明通风、显示系统、钢丝绳、导轨润滑。AI 辅助包括声音识别（异常频率）、视频识别（门缝异物）、外观识别（锈蚀/变形）、OCR（显示屏读数校验）。

**配电房检查清单（周检，8 项）：** 变压器温度、开关柜状态、接地系统、通风散热、消防设施、电缆桥架、防潮防鼠、运行声音。AI 辅助包括 OCR（温度表盘/灭火器有效期）、外观识别（指示灯/锈蚀/松动）、图像识别（变色/杂物/孔洞）。

**消防泵房检查清单（季检，8 项）：** 水泵运行、阀门状态、压力表读数、管道连接、应急电源、控制柜、水箱水位、管道标识。AI 辅助包括声音识别 + 振动分析、外观识别（泄漏/锈蚀/渗漏）、OCR（压力/电压读数）、图像识别（水位线/标识完整性）。

### 设备检查频率规范

| 设备类型 | 检查频率 | 检查深度 | 检查人员 |
|---------|---------|---------|---------|
| 电梯 | 月检 | 全面检查（8 项） | 技术员 + 电梯维保 |
| 配电房 | 周检 + 月检 | 周检：外观 + 温度；月检：全面 | 技术员 |
| 消防泵房 | 季检 | 全面检查（8 项） | 技术员 + 消防维保 |
| 空调系统 | 半年检 | 换季前全面检查 | 技术员 + 空调维保 |
| 照明系统 | 月检 | 区域抽检 | 巡检员 |
| 给排水 | 月检 | 管道 + 水泵检查 | 技术员 |

### ① 计划 — 设备检查计划编排与下发

| 维度 | 内容 |
|------|------|
| **角色** | 技术主管（陈志远）编排 → 技术员（张建国）接收 |
| **输入** | 设备台账、维保规范、历史检查数据、设备运行状态 |
| **输出** | 设备检查计划（含设备清单、检查项目、检查频率、预计时长） |
| **触发方式** | 系统按设备维保规范自动排期；设备告警触发临时检查 |

**流程描述：**

1. 系统根据设备台账和维保规范，自动生成周期性检查计划
2. 技术员在移动端查看本周/本月待检设备清单，按优先级排序
3. 技术主管可在 PC 后台调整检查计划（如新增临时检查、调整检查顺序）

**AI 嵌入：**

| AI 能力 | 具体作用 |
|--------|---------|
| 预测分析 | 基于设备历史故障数据，预测高风险设备，优先安排检查 |
| 智能排期 | 综合设备重要性、历史故障率、维保单位档期，自动生成最优检查排期 |

**GIS 嵌入：**

| GIS 能力 | 具体作用 |
|---------|---------|
| BIM 空间可视化 | 在 BIM 模型上展示待检设备位置，技术主管直观审视检查覆盖范围 |
| 设备分布分析 | 识别检查盲区——某楼层设备长期未检查，系统自动告警 |

**SLA：** 计划于检查日前 1 天完成编排下发；设备告警触发的临时检查 ≤2h 内响应。

### ② 执行 — 按检查清单逐项检查

| 维度 | 内容 |
|------|------|
| **角色** | 技术员（张建国） |
| **输入** | 设备检查计划 + 设备检查清单模板 |
| **输出** | 检查记录（含各检查项结果、现场照片、设备参数读数） |
| **触发方式** | 技术员到达设备点位后点击"开始检查" |

**流程描述：**

1. 技术员打开 APP，查看待检设备清单，点击"开始检查"
2. 系统展示该设备类型的检查清单模板
3. 技术员按清单逐项检查：外观类拍照留证；参数类拍摄表盘读数（OCR 自动识别数值）；功能类执行功能测试后记录结果
4. 全部检查项完成后，点击"提交检查记录"

**BIM/室内定位嵌入：**

| 定位能力 | 具体作用 |
|---------|---------|
| BIM 楼层定位 | 技术员进入楼层后，BIM 自动定位到当前楼层，展示该楼层所有待检设备位置 |
| 设备导航 | 在 BIM 模型上点击目标设备，系统规划室内导航路径 |
| 到位验证 | 技术员到达设备附近（蓝牙信标/二维码扫码），系统验证已到位后才允许开始检查 |

**SLA：** 单台设备检查需在计划时长内完成（如电梯检查 30min，配电房巡检 20min）。

### ③ 发现 — 设备问题发现与 AI 辅助判断

| 维度 | 内容 |
|------|------|
| **角色** | 技术员（张建国）发现 → AI 辅助判断严重程度 |
| **输入** | 检查过程中发现的异常（外观异常、参数超标、功能异常） |
| **输出** | 设备问题记录（含异常照片、OCR 读数、AI 判断结果、严重等级） |
| **触发方式** | 技术员在检查清单中标记某项"异常" |

**流程描述：**

1. 技术员在检查清单中将异常项标记为"异常"，系统弹出异常上报界面
2. 拍照（支持多角度多张）→ 录音（记录异常声音）→ 填写异常描述（或语音转文字）
3. AI 自动分析：外观照片识别缺陷类型；OCR 识别表盘读数并比对标准范围；录音分析异常频率
4. AI 综合判断严重等级并给出建议：紧急（P0，影响安全运行）、严重（P1，影响正常使用）、一般（P2，不影响运行）
5. 技术员确认 AI 判断结果，提交异常记录

**AI 嵌入：**

| AI 能力 | 具体作用 |
|--------|---------|
| 外观识别 | 自动识别缺陷类型：锈蚀、泄漏、变形、裂纹、磨损 |
| OCR 表盘识别 | 识别温度表、压力表、电压表、电流表读数，自动与标准范围比对，超标则告警 |
| 声音分析 | 录音后分析频谱，识别异常声音类型：异响、放电声、振动异常 |
| 综合判断 | 综合外观 + 参数 + 声音三项数据，AI 给出设备健康度评分（0-100 分）和严重等级建议 |

**GIS 嵌入：**

| GIS 能力 | 具体作用 |
|---------|---------|
| BIM 定位打标 | 异常设备在 BIM 模型上标记，管理层可直观查看异常设备分布 |
| 空间关联 | 自动关联设备所属楼层、区域、所属系统，便于统计分析 |

**SLA：** 异常发现到完成上报 ≤5min（拍照 + OCR + AI 判断 + 提交）。

### ④ 工单 — 设备维修工单创建与分派

| 维度 | 内容 |
|------|------|
| **角色** | 系统自动创建工单 → 技术主管（陈志远）审核分派 → 维保单位接单 |
| **输入** | 步骤③的异常记录（含 AI 判断结果） |
| **输出** | 已分派的设备维修工单（含维保单位、SLA 时限） |
| **触发方式** | 步骤③提交后自动触发 |

**流程描述：**

1. 系统自动创建设备维修工单，预填字段由前序步骤自动带入
2. 技术主管在 PC 后台审核工单：确认 AI 判断的严重等级、确认是否需要外部维保单位、设定 SLA 时限
3. 分派至对应维保单位或内部维修队
4. 维保单位收到 APP 推送，含工单详情、设备位置、BIM 导航入口

**AI 嵌入：**

| AI 能力 | 具体作用 |
|--------|---------|
| 智能推荐 | 基于设备类型 + 故障类型 + 历史合作记录，推荐最优维保单位（Top3 候选） |
| 方案推荐 | 匹配历史相似工单的维修方案，供技术主管参考 |
| SLA 建议 | 基于故障严重程度和设备重要性，自动建议 SLA 时限 |

**GIS 嵌入：**

| GIS 能力 | 具体作用 |
|---------|---------|
| BIM 设备定位 | 工单自动关联设备在 BIM 模型中的精确位置（楼层 + 房间 + 具体位置） |
| 就近匹配 | 基于维保单位服务范围和当前位置，推荐最近可用单位 |

**SLA：**
- 从异常上报到工单创建：≤1min（自动完成）
- 从工单创建到技术主管审核：隐患审核 ≤24h
- 紧急维修：P0/P1 工单 ≤48h 完成
- 一般维修：P2 工单 ≤7d 完成

### ⑤ 处置 — 设备维修处置

| 维度 | 内容 |
|------|------|
| **角色** | 维保单位/内部维修队 |
| **输入** | 已分派的设备维修工单 |
| **输出** | 维修完成记录（维修报告、更换部件清单、维修后照片、测试数据） |
| **触发方式** | 维保单位到场后点击"开始维修" |

**流程描述：**

1. 维保单位通过 BIM 导航至设备位置，到达后点击"开始维修"
2. 维修过程中：查看 AI 推荐的维修方案和历史相似案例；拍照记录维修过程；填写维修报告
3. 维修完成后进行功能测试（电梯试运行、配电房参数测量、消防泵房启动测试）
4. 上传维修报告 + 测试数据 + 维修后照片 → 提交

**BIM/室内定位嵌入：**

| 定位能力 | 具体作用 |
|---------|---------|
| BIM 室内导航 | 从进入大楼开始，BIM 提供逐层导航至目标设备 |
| 到位验证 | 通过蓝牙信标/二维码扫码验证已到位，防止未到场虚报 |
| 维修区域标注 | 在 BIM 模型上标注正在维修的设备，提醒其他人员注意安全 |

**SLA：**
- P0/P1 工单：从分派到维修完成 ≤48h
- P2 工单：从分派到维修完成 ≤7d
- 超时升级规则：超时 24h → 通知技术主管；超时 48h → 通知物业经理

### ⑥ 技术验收 — 专业技术人员验收

| 维度 | 内容 |
|------|------|
| **角色** | 技术主管（陈志远）或指定技术专家 |
| **输入** | 维修完成记录（维修报告、测试数据、维修后照片） |
| **输出** | 验收结果（通过/不通过）+ 验收报告 |
| **触发方式** | 维修提交后自动触发验收流程 |

**流程描述：**

1. 系统自动推送验收任务至技术主管
2. 技术主管在 PC 后台审核维修报告：检查维修措施是否针对故障根因、更换部件是否符合规格、测试数据是否达标
3. **远程审核**：查看维修照片 + 测试数据 + AI 图像比对结果
4. **现场验收**（必要时）：技术主管到达设备现场，独立进行功能测试，拍照记录验收状态
5. 验收通过 → 工单闭环；验收不通过 → 工单退回维保单位，标注不通过原因，重新维修

**与日常巡检验收的核心差异：**

| 维度 | 日常巡检验收 | 设备技术验收 |
|------|-------------|-------------|
| **验收人员** | 巡检员或管理层 | 技术主管或指定技术专家 |
| **验收内容** | 照片比对（处置前后） | 功能测试 + 参数验证 + 照片比对 |
| **验收标准** | 问题是否消失 | 设备功能是否恢复正常 + 参数是否达标 |
| **验收深度** | 表面确认 | 深度技术验证（可能需要现场） |

**AI 嵌入：**

| AI 能力 | 具体作用 |
|--------|---------|
| 图像比对 | 自动对比"上报时照片"与"维修后照片"，判断外观缺陷是否修复 |
| 参数验证 | OCR 读取维修后测试数据，自动比对标准范围，判断参数是否达标 |
| 质量评分 | 综合图像比对 + 参数验证 + 维修报告完整性，给出维修质量评分（1-5 分） |

**GIS 嵌入：**

| GIS 能力 | 具体作用 |
|---------|---------|
| BIM 现场定位 | 技术主管现场验收时，BIM 定位验证是否在设备附近 |
| 验收状态标注 | 验收通过的设备在 BIM 模型上恢复正常状态标识 |

**SLA：**
- 远程审核：维修完成后 ≤24h 内完成
- 现场验收：维修完成后 ≤48h 内完成
- 验收不通过退回重维修：SLA 重新计算

### ⑦ 分析 — 设备健康度分析与决策优化

| 维度 | 内容 |
|------|------|
| **角色** | 技术主管（陈志远）查看 → 系统自动产出分析报告 |
| **输入** | 全流程数据（检查记录、异常数据、工单、维修、验收） |
| **输出** | 设备健康度报告、维保建议、设备更新建议 |
| **触发方式** | 每月自动生成设备健康度报告；重大故障触发专项分析 |

**流程描述：**

1. 系统每月 1 日自动生成设备健康度月报
2. 月报核心指标：设备完好率、故障率、平均故障修复时间（MTTR）、维修工单闭环率、SLA 达成率
3. 技术主管在 PC 后台查看可视化报表，支持按设备类型/楼层/系统下钻
4. 系统自动产出决策建议：预防性维保建议、设备更新建议、维保单位绩效评估

**AI 嵌入：**

| AI 能力 | 具体作用 |
|--------|---------|
| 设备劣化预测 | 基于历史检查数据（OCR 读数趋势、外观缺陷频率），预测设备劣化曲线，提前预警 |
| 异常模式识别 | 识别设备异常模式：某类设备集中故障、某维保单位维修质量下降 |
| 优化建议 | 自动生成维保策略优化建议（如检查频率调整、维保单位更换） |

**GIS 嵌入：**

| GIS 能力 | 具体作用 |
|---------|---------|
| BIM 热力图 | 在 BIM 模型上生成设备故障热力图，识别故障高发楼层/区域 |
| 设备生命周期 | 在 BIM 模型上展示设备年龄、维修历史、健康度评分，支持设备更新决策 |

**SLA：** 月报于每月 3 日前自动生成；重大故障专项分析 ≤72h。

### 数据回流闭环

分析（⑦）产出的优化建议，经技术主管确认后回流至计划（①）：

```
⑦分析 → 优化建议 → 技术主管确认 → ①计划自动更新 → ②执行调整后的检查
```

| 调整类型 | 触发条件 | 示例 |
|---------|---------|------|
| **频率调整** | 设备故障率上升（升频）或持续稳定（降频） | C 区电梯故障率上升→月检改为双周检 |
| **检查深度调整** | 某类设备频繁出现同类故障 | 消防泵房阀门频繁泄漏→检查清单增加专项检查 |
| **设备更新** | 设备健康度持续低于阈值 | 某电梯健康度评分连续 3 个月 <60 分→建议大修或更换 |
| **维保单位调整** | 维保单位维修质量评分下降 | 连续 2 次验收不通过→更换维保单位 |

### SLA 总览

| 阶段 | SLA 时限 | 升级规则 |
|------|---------|---------|
| ① 计划下发 | 检查日前 1 天完成 | 未下发→自动告警技术主管 |
| ② 检查执行 | 计划时长内完成 | 超时→告警技术主管 |
| ③ 异常上报 | ≤5min | — |
| ④ 隐患审核 | ≤24h | 超时→通知技术主管 |
| ⑤ 紧急维修（P0/P1） | ≤48h | 超时 24h→技术主管；超时 48h→物业经理 |
| ⑤ 一般维修（P2） | ≤7d | 超时→提醒物业经理 |
| ⑥ 技术验收 | 远程≤24h / 现场≤48h | 验收不通过→退回重维修，SLA 重计 |
| ⑦ 分析报告 | 月报每月 3 日前 | — |

## 六、场景③：应急事件全流程

### 6.1 应急场景定义

应急事件是指园区内突然发生、需立即响应处置的紧急情况：

| 事件类型 | 典型场景 | 危害等级 | 主要处置角色 |
|---------|---------|---------|------------|
| **火灾** | 电气短路、明火、烟雾 | P0 | 安保 + 消防 + 全员疏散 |
| **水管爆裂** | 主水管破裂、阀门失效 | P0 | 维修队 + 安保 |
| **电梯困人** | 电梯故障、停电 | P0 | 维保单位 + 安保 |
| **人员受伤** | 滑倒、高空坠物 | P0/P1 | 安保 + 医疗 + 管理层 |
| **设备重大故障** | 配电房异常、空调系统瘫痪 | P1 | 维修队 + 管理层 |

### 6.2 应急独立 SLA（区别于日常巡检）

| 阶段 | SLA 时限 | 说明 |
|------|--------|------|
| **确认** | ≤3min | IoT + 人工双确认事件真实性 |
| **到场** | ≤10min | 处置人员到达现场 |
| **响应** | ≤10min | 指挥中心启动预案、调度资源 |

> 应急 SLA 独立于日常巡检 SLA（日常巡检为 P2/P3 级，响应 ≤2h/24h），应急事件为 P0/P1 级，要求分钟级响应。

### 6.3 应急优先级定义

| 级别 | 定义 | 响应方式 | 升级规则 |
|------|------|---------|---------|
| **P0** | 重大安全事件，需立即响应 | 全员推送 + 指挥中心接管 + 并行响应 | 3min 未确认 → 自动升级至最高级；连续 2 次不确认 → 强制升级 |
| **P1** | 严重隐患，需尽快处理 | 相关角色推送 + 指挥中心协调 | 10min 未响应 → 升级至 P0 |

### 6.4 应急全流程（七步闭环）

应急流程采用**非线性、并行响应**模式，与日常巡检的线性流程有本质区别：

```
触发 → 确认 → 研判调度 → 并行响应 → 态势追踪 → 解除 → 复盘
```

#### ① 触发（事件发生）

| 触发方式 | 具体场景 | 系统响应 |
|---------|---------|---------|
| **IoT 告警** | 烟雾传感器报警、水浸传感器触发、电梯故障信号 | 系统自动接收告警，立即启动应急流程 |
| **人工紧急上报** | 巡检员发现火情、员工报告水管爆裂 | 通过 App 一键上报，系统立即启动应急流程 |
| **系统检测** | 视频 AI 识别到异常行为、设备参数超阈值 | 系统自动分析，触发应急流程 |

**触发规则：** 任何单一触发源均启动应急流程，不等待多源确认（确认环节并行进行）。触发后系统自动记录：时间、位置、触发源、初始描述。

#### ② 确认（IoT + 人工双确认，≤3min）

确认环节采用并行核查机制，多源数据同时验证：

| 确认方式 | 执行者 | 动作 | 时限 |
|---------|--------|------|------|
| **IoT 传感器核查** | 系统自动 | 读取相关传感器数据（温度、水位、烟雾浓度等） | ≤30s |
| **视频联动核查** | 系统自动 | 调取最近摄像头画面，AI 分析是否匹配告警类型 | ≤1min |
| **就近人员现场确认** | 最近可用人员 | 接收推送，前往现场确认，拍照/视频上传 | ≤3min |

**确认逻辑：** IoT + 视频任一确认即视为有效告警。人工确认为最终确认，如与 IoT/视频不一致，以人工确认为准。3min 内无任何确认 → 自动升级至最高级 P0，全员推送。

#### ③ 研判调度（指挥中心接管）

确认后，指挥中心立即接管，进行事件定级与资源调度：

| 步骤 | 执行者 | 动作 | 输出 |
|------|--------|------|------|
| **事件定级** | 指挥中心 + AI | 基于事件类型、影响范围、危害程度确定 P0/P1 级别 | 事件级别、响应范围 |
| **GIS 缓冲区分析** | 系统自动 | 以事件点为中心，分析影响半径 | 影响范围、需疏散区域、可用资源分布 |
| **AI 预案匹配** | AI 引擎 | 匹配历史类似事件处置预案，推荐最佳响应方案 | 预案推荐、处置步骤、所需资源 |
| **资源调度** | 指挥中心 | 基于 GIS 就近分析，调度最近可用人员、设备、物资 | 调度指令、责任人、预计到场时间 |

**调度规则：** P0 事件指挥中心直接接管，所有角色立即响应。GIS 缓冲区分析半径：火灾 50m、水管爆裂 30m、电梯困人 10m（可配置）。

#### ④ 并行响应（多角色同时执行）

应急响应采用多线程并行模式，各角色同时执行不同任务：

| 线程 | 角色 | 任务 | 与其他线程关系 |
|------|------|------|--------------|
| **线程 1：现场处置** | 巡检员/维修队 | 到达现场，执行处置操作（灭火、关阀、急救等） | 主线程，其他线程配合 |
| **线程 2：设备抢修** | 维保团队 | 设备故障抢修（电梯、水管、电气等） | 与线程 1 并行，技术处置 |
| **线程 3：疏散引导** | 安保人员 | 疏散周边人员，设置警戒线，引导至安全区域 | 与线程 1 并行，安全保障 |
| **线程 4：资源协调** | 管理层 | 协调外部资源（消防、医疗、供应商），信息上报 | 支持线程，资源保障 |

```
指挥中心（态势地图）
    ├── 线程 1：现场处置（巡检员/维修队）
    ├── 线程 2：设备抢修（维保团队）
    ├── 线程 3：疏散引导（安保人员）
    └── 线程 4：资源协调（管理层）
```

**并行规则：** 各线程独立执行，通过指挥中心协调。线程间信息实时同步。任一线程完成任务后，向指挥中心报告，等待指令。

**线程间协同：**

| 协同场景 | 处理方式 |
|---------|---------|
| **资源冲突** | 指挥中心统一调度，优先级高的线程优先使用资源 |
| **信息同步** | 各线程通过指挥中心中转，避免直接通信混乱 |
| **任务依赖** | 如疏散完成是现场处置的前提，指挥中心协调时序 |
| **异常处理** | 任一线程异常，指挥中心立即调整其他线程任务 |

#### ⑤ 态势追踪（指挥中心实时追踪）

指挥中心通过态势地图实时监控各线程状态：

| 监控维度 | 显示内容 | 更新频率 |
|---------|---------|---------|
| **事件位置** | 事件点位、影响范围（GIS 缓冲区） | 实时 |
| **人员分布** | 各线程人员位置、移动轨迹 | 每 10s |
| **资源状态** | 设备、物资使用情况 | 每 30s |
| **处置进展** | 各线程任务完成度、关键节点 | 事件驱动 |
| **时间线** | 各阶段耗时、SLA 达成情况 | 实时 |

**态势地图功能：** GIS 多图层叠加（事件点、影响范围、人员位置、资源分布）、实时通信、预警提示（SLA 即将超时、资源不足等）、历史回放。

#### ⑥ 解除（各线程逐一确认 → 恢复常态）

| 步骤 | 执行者 | 动作 | 确认标准 |
|------|--------|------|---------|
| **现场处置确认** | 巡检员/维修队 | 确认处置完成，拍照上传 | 问题已解决，现场安全 |
| **设备恢复确认** | 维保团队 | 确认设备恢复正常运行 | 设备参数正常，无异常 |
| **人员清点确认** | 安保人员 | 确认疏散人员已清点，无遗漏 | 人员全部安全，警戒解除 |
| **指挥中心确认** | 管理层 | 综合各线程报告，确认事件解除 | 所有线程确认完成，无遗留风险 |

**解除规则：** 所有线程逐一确认后，指挥中心方可宣布事件解除。解除后系统自动恢复常态巡检计划。解除信息全员推送。

#### ⑦ 复盘（AI 自动生成复盘报告 + 预案优化建议）

| 复盘内容 | 生成方式 | 输出 |
|---------|---------|------|
| **事件时间线** | 系统自动记录各节点时间 | 从触发到解除的完整时间线 |
| **SLA 达成分析** | 系统自动计算各阶段耗时 | 确认、到场、响应 SLA 达成情况 |
| **资源使用分析** | 系统记录资源调度情况 | 人员、设备、物资使用效率 |
| **处置效果评估** | AI 分析处置前后对比 | 处置效果评分、改进建议 |
| **预案优化建议** | AI 对比历史类似事件 | 预案优化点、流程改进建议 |

**复盘输出：** 自动生成《应急事件复盘报告》，自动更新应急预案库并优化 AI 预案匹配模型，自动调整相关区域巡检频率。

### 6.5 AI 与 GIS 嵌入点

**AI 嵌入点：**

| 环节 | AI 能力 | 具体作用 |
|------|--------|---------|
| 确认 | 图像识别 + 视频分析 | 视频联动核查，自动判断告警真实性 |
| 研判调度 | 预案匹配 | 基于历史数据匹配最佳处置预案 |
| 并行响应 | 知识库推荐 | 为各线程推荐处置方案、操作步骤 |
| 态势地图 | 异常检测 | 实时监测各线程状态，自动预警异常 |
| 复盘 | 报告生成 + 优化建议 | 自动生成复盘报告，提出预案优化建议 |

**GIS 嵌入点：**

| 环节 | GIS 能力 | 具体作用 |
|------|---------|---------|
| 确认 | 定位打标 | 事件点位精确定位，自动标注到地图 |
| 研判调度 | 缓冲区分析 | 分析事件影响范围，确定疏散区域、可用资源 |
| 并行响应 | 就近匹配 + 导航 | 调度最近可用人员，导航至事件点位 |
| 态势地图 | 多图层叠加 | 事件点、影响范围、人员位置、资源分布实时叠加 |
| 复盘 | 空间分析 | 分析事件空间分布规律，优化应急预案 |

### 6.6 跨场景联动

**从其他场景触发应急：**

| 触发源 | 场景 | 联动方式 |
|--------|------|---------|
| **日常巡检发现重大隐患** | 日常巡检 → 应急事件 | 巡检员上报，系统自动触发应急流程 |
| **设备检查发现重大故障** | 设备检查 → 应急事件 | 设备参数超阈值，系统自动触发应急流程 |
| **IoT 持续告警** | 设备监控 → 应急事件 | 设备连续异常，系统自动升级为应急事件 |

**应急事件反向影响其他场景：**

| 影响方向 | 具体影响 |
|---------|---------|
| **应急 → 日常巡检** | 事件处置完成后，自动提升相关区域巡检频率 |
| **应急 → 设备检查** | 设备故障事件后，自动安排专项设备检查 |
| **应急 → 预案优化** | 复盘报告自动更新应急预案库，优化 AI 匹配模型 |

**数据回流：** 应急事件数据自动进入分析引擎，驱动事件聚类分析、SLA 达成分析、预案效果评估、资源使用优化。

### 6.7 应急流程示例：水管爆裂

**事件背景：** B 区 3 楼主水管爆裂，大量漏水，影响办公区域。

| 阶段 | 时间 | 动作 | 负责方 |
|------|------|------|--------|
| **触发** | T+0s | 水浸传感器报警 + 巡检员发现漏水上报 | 系统 + 巡检员 |
| **确认** | T+30s | IoT 读取水浸传感器数据，确认漏水 | 系统自动 |
| | T+1min | 视频联动，调取 B 区 3 楼摄像头，AI 确认大量漏水 | 系统自动 |
| | T+2min | 就近巡检员到达现场，拍照确认水管爆裂 | 巡检员 |
| **研判调度** | T+3min | 指挥中心定级 P0，启动应急响应 | 指挥中心 |
| | T+3.5min | GIS 缓冲区分析：影响半径 30m，涉及 B 区 3-4 楼 | 系统自动 |
| | T+4min | AI 匹配预案：关闭上游阀门 + 维修 + 疏散 + 清理 | AI 引擎 |
| | T+4.5min | 调度：维修队（最近 2 人）、安保（最近 3 人）、保洁（最近 2 人） | 指挥中心 |
| **并行响应** | T+5min | 线程 1：维修队到达，关闭上游阀门 | 维修队 |
| | T+5min | 线程 2：安保疏散 B 区 3-4 楼人员，设置警戒线 | 安保 |
| | T+5min | 线程 3：管理层协调外部资源（如需） | 管理层 |
| | T+15min | 维修队完成临时修复，漏水停止 | 维修队 |
| | T+20min | 安保确认人员疏散完成，无遗漏 | 安保 |
| **态势追踪** | T+0~25min | 指挥中心通过态势地图实时监控各线程状态 | 指挥中心 |
| **解除** | T+25min | 现场处置确认、设备恢复确认、人员清点确认 | 各线程 |
| | T+26min | 指挥中心确认事件解除，恢复常态 | 指挥中心 |
| **复盘** | T+30min | AI 自动生成复盘报告：从发现到解除耗时 26min，SLA 达成 | 系统自动 |

**SLA 达成情况：**

| 阶段 | SLA 要求 | 实际耗时 | 达成情况 |
|------|--------|---------|---------|
| 确认 | ≤3min | 2min | 达成 |
| 到场 | ≤10min | 5min | 达成 |
| 响应 | ≤10min | 4.5min | 达成 |

### 6.8 关键设计决策

1. **非线性并行响应**：应急流程是多角色多线程并行，通过指挥中心协调，确保响应速度最大化。
2. **IoT + 人工双确认**：IoT 传感器、视频 AI、人工现场确认三重验证，确保告警真实性。
3. **指挥中心统一调度**：应急事件由指挥中心接管，避免多头指挥。
4. **态势地图实时追踪**：指挥中心通过态势地图实时掌握各线程状态。
5. **独立 SLA 体系**：应急 SLA 与日常巡检 SLA 分离，确保分钟级响应。
6. **AI 深度参与**：从预案匹配到复盘优化，AI 贯穿应急全流程。
7. **数据驱动自优化**：应急事件数据自动回流分析引擎，驱动预案优化、巡检调整、资源布局优化。

| **应急响应启动** | — | — | ≤10min | 确认后指挥中心接管 |
| **应急到场** | — | — | ≤10min | 从触发到处置人到达 |

> **统一原则**：若同一工单同时满足多个 SLA 条件，以最短时限为准；所有时限均从上一节点完成时刻开始计算。

### 7.2 升级策略

升级分为三类：超时自动升级、严重性自动升级、L3 最高级响应。三类升级可叠加触发。

**超时自动升级：**

| 超时节点 | 首次升级（T+30min） | 二次升级（T+1h） | 三次升级（T+2h） |
|---------|-------------------|-----------------|-----------------|
| **工单审核/分派** | 通知主管 | 通知经理 | 通知园区负责人 |
| **处置人到场** | 通知主管 | 通知经理 | 通知园区负责人 |
| **处置完成** | 通知主管 | 通知经理 | 通知园区负责人 |
| **应急确认** | 3min 未确认 → 自动升至 P0 最高级 | — | — |
| **应急到场** | 10min 未到场 → 升级调度备选人员 | — | — |

> 特殊规则：应急事件连续 2 次推送不确认 → 强制升级至最高级，全员推送 + 指挥中心接管。

**严重性自动升级：**

| 触发条件 | 升级动作 | 执行者 |
|---------|---------|--------|
| AI 判定严重等级为 P0（重大安全） | 自动升级为紧急工单，跳过常规审核 | 系统 |
| 问题类型为消防类（火灾/消防通道堵塞/消防设备失效） | 自动升级至最高级响应 | 系统 |
| 同类问题单周出现 ≥3 次 | 自动标记为"重点关注"，升级至经理层面 | 系统 |
| 设备参数持续超阈值（连续 3 次检测超标） | 自动升级为 P1，触发专项设备检查 | 系统 |
| 日常巡检发现设备异常 → 自动触发设备检查 | 按设备检查 SLA 执行 | 系统 |
| 设备检查发现重大隐患 → 自动触发应急事件 | 按应急 SLA 执行 | 系统 |

**L3 最高级响应（P0 应急/重大安全事件）：**

| 响应维度 | 具体动作 |
|---------|---------|
| **通知范围** | 全员推送（巡检员、管理层、指挥中心、安保、维保单位） |
| **指挥模式** | 指挥中心接管，统一调度所有资源 |
| **通信机制** | 主管 + 经理自动拉群（企业微信/钉钉），实时同步事件进展 |
| **处置超时** | 若处置超时仍未解决，自动上报园区负责人 |
| **资源权限** | 指挥中心可跨部门调度人员、设备、物资，无需逐级审批 |
| **信息同步** | 态势地图实时同步至所有相关人员，确保信息透明 |

### 7.3 通知升级链路

```
T+0          → 巡检员/处置人（首次通知）
T+30min      → 主管（首次升级）
T+1h         → 经理（二次升级）
T+2h         → 园区负责人（三次升级）
```

**通知渠道优先级：**

| 优先级 | 渠道 | 适用场景 |
|--------|------|---------|
| P0 | APP 推送 + 短信 + 电话 | P0 应急事件、L3 最高级响应 |
| P1 | APP 推送 + 短信 | P1 严重工单、超时升级 |
| P2 | APP 推送 | P2 一般工单 |
| P3 | APP 推送（每日汇总） | P3 轻微问题（批量通知） |

**通知静默规则：**
- 夜间（22:00-07:00）非应急事件不触发短信/电话，仅 APP 推送（次日 07:00 批量提醒）
- 同一工单的同一升级节点，24h 内不重复通知
- 已闭环工单不再触发任何通知

### 7.4 优先级策略

系统采用 P0-P3 四级优先级体系，覆盖所有场景：

| 级别 | 判定条件 | SLA 响应 | SLA 处置 | 适用场景 |
|------|---------|---------|---------|---------|
| **P0** | 重大安全事件；火灾、电梯困人、水管爆裂、人员伤亡 | ≤3min 确认 / ≤10min 到场 | 立即处置，直至解除 | 应急事件 |
| **P1** | 严重隐患；设备重大故障、消防通道堵塞、电气裸露、结构损坏 | ≤30min 到场 | ≤2h 完成 / 设备≤48h | 设备故障、安全隐患 |
| **P2** | 一般问题；照明损坏、墙面污损、设备表面锈蚀、标识脱落 | ≤2h 到场 | ≤24h 完成 / 设备≤7d | 日常巡检一般问题 |
| **P3** | 轻微问题；地面纸屑、绿化落叶、椅子未归位、标识歪斜 | 当场处置 | ≤10min 闭环 | 日常巡检轻微问题 |

**优先级动态调整：**

| 调整场景 | 规则 |
|---------|------|
| 同类问题周频次≥3 | P2 自动升为 P1 |
| 设备参数连续 3 次超标 | P2 自动升为 P1 |
| 消防类问题 | 自动升为 P1（一般）或 P0（严重） |
| 应急事件处置超时 | 维持 P0，触发 L3 最高级响应 |
| 工单被退回 2 次 | 自动升一级优先级 |

**优先级与工单的绑定规则：**
- 优先级在工单创建时由 AI 辅助判定，管理层可手动调整
- 优先级一旦调整，SLA 重新计算
- 同一工单不支持多优先级，以当前最高优先级为准

### 7.5 Edge Cases

#### 7.5.1 离线场景

**问题定义：** 巡检员在地下室、电梯轿厢、偏远区域等无网络环境下执行巡检或上报问题。

**影响分析：** 无法实时同步巡检轨迹和上报数据；GPS 定位精度下降或失效；工单状态无法实时更新；管理层无法查看实时进度。

**处理原则：**
- 移动端支持离线缓存，巡检数据本地存储
- 恢复网络后自动断网续传，按时间戳排序同步
- 离线期间的上报记录标记"离线提交"，同步后由 AI 重新判定优先级
- 地下室等场景切换至 BIM/蓝牙信标/二维码定位作为 GPS 补充

#### 7.5.2 误报处理

**问题定义：** AI 图像识别将正常状态误判为异常（如反光误判为积水、阴影误判为裂缝），或巡检员误操作上报。

**影响分析：** 产生无效工单，浪费处置资源；降低 AI 可信度，影响巡检员使用意愿；误报率过高会导致"狼来了"效应，延误真实问题处理。

**处理原则：**
- AI 识别置信度 < 85% 时强制人工复核，不自动创建工单
- 所有 AI 自动判定的工单必须经管理层或巡检员确认后才进入分派流程
- 建立误报反馈机制：处置人可将工单标记为"误报"，系统自动记录并反馈至 AI 模型
- 月度统计误报率，若某类问题误报率 > 20%，暂停该类自动识别，改为全人工复核

#### 7.5.3 多园区扩展

**问题定义：** V1.0 仅支持单园区，未来 V2.0 需扩展至多租户、多园区部署。

**影响分析：** 数据结构需支持园区隔离；管理层权限需按园区划分；分析引擎需支持跨园区对比；工单流转不能跨园区。

**处理原则（V1.0 预埋）：**
- 数据库设计预留 `park_id` 字段，为 V2.0 多租户做准备
- 用户权限模型预留园区维度，V1.0 默认单园区
- API 接口预留园区参数，V1.0 传默认值
- V1.0 不做多园区 UI 切换，但数据层不阻塞扩展

#### 7.5.4 权限冲突

**问题定义：** 多人同时上报同一问题、多人同时处置同一工单、管理层与巡检员同时修改工单状态。

**影响分析：** 重复工单导致资源浪费；并发修改导致数据不一致；工单状态混乱。

**处理原则：**
- 同一点位 10min 内多次上报，系统自动合并为同一工单，保留最早记录
- 工单状态机设计为严格的单向流转（新建 → 分派中 → 处置中 → 待验收 → 已闭环/已退回），禁止逆向跳转
- 状态变更需校验当前状态，冲突时后提交者提示"工单状态已变更，请刷新"
- 管理层修改工单状态时，若处置人正在编辑，锁定工单 30 秒

#### 7.5.5 数据隐私

**问题定义：** 巡检轨迹、人员位置、问题照片等数据涉及员工隐私和园区安全信息。

**影响分析：** 巡检轨迹可还原个人行动路线，存在隐私泄露风险；问题照片可能包含敏感区域；管理层可查看全员实时位置，需防止越权访问。

**处理原则：**
- 巡检轨迹数据保留 90 天，过期自动脱敏（仅保留区域级热力图，删除个人轨迹）
- 问题照片按权限分级：巡检员可见自己上报的照片；管理层可见管辖区域照片；涉及机房等敏感区域的照片仅技术主管可见
- 实时位置仅指挥中心和管理层在应急事件中可查看，日常巡检仅展示"巡检中/离线"状态
- 所有数据访问记录审计日志，保留 1 年

#### 7.5.6 设备兼容性

**问题定义：** 巡检员使用的移动设备型号各异，系统需保证在不同设备上的可用性。

**影响分析：** 低端设备性能不足，AI 图像识别卡顿或闪退；屏幕尺寸差异导致 UI 布局错乱；旧版本操作系统不支持新 API；不同浏览器对 GIS 地图渲染效果不一致。

**处理原则：**
- 移动端采用响应式设计，适配 5.5-6.7 英寸主流屏幕
- 最低版本要求：Android 8.0+ / iOS 13+，低于此版本提示升级
- PC 后台支持 Chrome 90+、Edge 90+、Firefox 88+，IE 不支持
- AI 图像识别在低端设备上降级为云端识别（拍照上传后云端返回结果），本地仅做缓存
- GIS 地图在弱网环境下降级为简化地图（减少图层和标注数量）

### 7.6 SLA 与 Edge Cases 关联矩阵

| Edge Case | 主要影响 SLA 节点 | 应对策略 |
|-----------|------------------|---------|
| 离线场景 | 上报同步、轨迹同步、工单状态更新 | 离线缓存 + 断网续传 + 降级定位 |
| 误报处理 | 工单创建、处置资源浪费 | 置信度阈值 + 人工复核节点 + 误报反馈闭环 |
| 多园区扩展 | 数据分析、权限隔离 | V1.0 数据层预埋，V2.0 启用多租户 |
| 权限冲突 | 工单状态一致性、重复工单 | 状态机单向流转 + 合并机制 + 乐观锁 |
| 数据隐私 | 轨迹查询、照片查看、位置追踪 | 分级权限 + 数据脱敏 + 审计日志 |
| 设备兼容性 | AI 响应速度、地图渲染、UI 可用性 | 响应式 + 版本底线 + 云端降级 |

|------|------|---------|
| **Phase 1：架构重构** | 第 1-2 个月 | 单体拆微服务、多租户数据隔离、权限体系重构 |
| **Phase 2：3D/BIM** | 第 2-4 个月 | 3D 园区建模、BIM 引擎集成、3D 可视化底座 |
| **Phase 3：场景扩展** | 第 4-6 个月 | 能耗/停车/安防场景接入，IoT 全量接入 |
| **Phase 4：数字孪生** | 第 6-9 个月 | 实时数据映射、仿真引擎、预测性维护 |
| **Phase 5：平台化** | 第 9-12 个月 | 开放 API、SaaS 化运营、多园区标杆客户落地 |

> [假设] V2.0 时间线跨度较大，需根据 V1.0 市场反馈和技术验证结果动态调整优先级。

### 8.5 版本决策依据

**为什么 V1.0 不做多租户？**
- **验证优先**：单园区的完整闭环已足够验证产品价值（SLA 驱动、AI 辅助、GIS 可视化）
- **复杂度控制**：多租户涉及数据库隔离、权限重构、配置化管理，会将 V1.0 工期拉长 50% 以上
- **面试展示**：PM 作品集的核心是「产品设计完整性」，而非「技术架构先进性」；单园区深度闭环比多园区浅层覆盖更有说服力

**为什么 V1.0 不做 3D/数字孪生？**
- **成本权衡**：3D 建模和数字孪生底座需额外 2-3 个月，且需专业 3D 美术和引擎开发资源
- **需求匹配**：V1.0 目标用户的核心痛点是「问题闭环慢、巡检效率低」，2D 地图 + SLA 驱动已能显著改善
- **演进路径**：V1.0 的 2D GIS 数据和 BIM 楼层定位数据可为 V2.0 的 3D 底座复用，不阻塞未来扩展

**为什么 V2.0 才做扩展场景？**
- **主线聚焦**：能耗、停车与巡检主线关联弱，独立场景需独立硬件接入和数据源
- **资源聚焦**：V1.0 团队规模有限，集中资源打磨 3 个核心场景的闭环体验
- **市场验证**：V1.0 落地后收集实际使用数据，再决定扩展场景的优先级排序

### 8.6 版本对照总表

| 维度 | V1.0 | V2.0 |
|------|------|------|
| 园区范围 | 单园区 | 多租户/多园区 |
| 地图 | 2D | 3D/BIM |
| 场景 | 日常巡检、设备检查、应急事件 | + 能耗、停车、安防 |
| AI | 图像识别、NLP、预测分析 | + 语音交互、知识图谱、数字孪生推理 |
| GIS | 可视化、聚类、缓冲区 | + 3D 漫游、BIM 联动、实时仿真 |
| 部署 | 单体 | 微服务 |
| 目标 | MVP 验证 | 平台规模化 |
| 工期 | ~3 个月 | ~6-12 个月 |
